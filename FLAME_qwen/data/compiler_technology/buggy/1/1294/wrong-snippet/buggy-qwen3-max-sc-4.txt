Based on the problem description and code review, the main issue appears to be in the error handling for missing semicolons (error type 'i'). The problem is that when a semicolon is missing, the parser looks back 2 tokens and reports the error on the line of the token before the expected semicolon, but according to the specification, "The error line number corresponds to the line where the last non-terminal before the semicolon is located."

Looking at multiple statement parsers, they all follow the same pattern for handling missing semicolons. The issue is in how they determine which line number to report for the missing semicolon error.
