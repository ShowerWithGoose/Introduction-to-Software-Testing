Based on the problem description and code review, the key issue appears to be in the error checking logic for missing semicolons (error type 'i'). The `checkI` method in `ErrorTool.java` is being called with the current token, but it should be checking for the presence of a semicolon after parsing a statement or declaration.

The main problem is in the parser methods where `checkI` is called - they pass the current token which may not be the semicolon yet, since `getToken()` hasn't been called after parsing the construct. This causes incorrect error reporting.

Looking at the code structure and the error handling pattern, the most likely location of the error is in the `parseConstDecl`, `parseVarDecl`, and `parseStmt` methods where semicolon checking occurs.
