Based on the problem description and code review, the key issue appears to be in the error handling for missing semicolons (error type 'i'). The `checkI` method in `ErrorTool.java` is being called with the current token, but it should be checking if the current token is a semicolon. However, looking at the parser code, particularly in methods like `parseConstDecl`, `parseVarDecl`, etc., the error checking happens after parsing the declaration/definition but before consuming the semicolon token.

The main problem is in the parser's handling of semicolon checking - it calls `errorTool.checkI(token, line)` where `token` is the current token (which should be the semicolon), but the logic is inverted. The `checkI` method returns true only if the token is ";", but in the parser code, it only consumes the next token if `checkI` returns true. This means if there's no semicolon, it won't consume anything and will get stuck.

Looking at the `parseConstDecl` method as an example:
