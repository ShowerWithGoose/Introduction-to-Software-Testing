Based on the problem description and code review, the key issue appears to be in the error handling for missing semicolons and parentheses. The test case input shows a compilation error related to javac not being found, but the actual bug is likely in the parser's error reporting logic.

The main problem is in the `lexer.special()` method which is used to determine the line number for error reporting. This method skips tokens marked as FAKE, but the FAKE tokens are inserted incorrectly in the parser, causing the line number calculation to be wrong.

Looking at the `fakeInsert` method in Parser.java:

<frontend/Parser.java>
...
    public void fakeInsert(String s){
        Token tok = new Token(s,LexType.FAKE,-1);
        tokens.add(tokens.get(tokens.size()-1));
        tokens.set(tokens.size()-2,tok);
    }
...

This method incorrectly manipulates the tokens array by adding and replacing elements, which breaks the correspondence between tokens and their original line numbers. This affects the `lexer.special()` method which tries to find the last non-FAKE token's line number.

The error is also evident in how semicolon errors are handled in multiple places like ConstDecl(), VarDecl(), etc., where `lexer.special()` is used to get the line number, but due to the incorrect token manipulation, it may not return the correct line.

However, the most direct issue is in the `fakeInsert` method itself, which corrupts the token stream structure needed for proper error line number reporting.
