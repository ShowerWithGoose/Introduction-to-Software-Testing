Based on the problem description and code review, the key issue appears to be in the error handling for missing semicolons (error type 'i'). The problem is that the error reporting uses the current token's line number, but according to the specification, "The error line number corresponds to the line where the last non-terminal before the semicolon is located."

Looking at the `handleErrorIJK` method and its usage in various parsing methods, the error is being reported at the current token position rather than at the position of the last successfully parsed non-terminal component.

The most likely location of the error is in the `Parser.java` file, specifically in how the error position is determined for missing semicolons.
