Based on the problem description and code review, the main issue appears to be in the **AddExp parsing logic** in the Parser.java file. The recursive descent implementation for AddExp (and similarly for other expression types like MulExp, RelExp, etc.) incorrectly outputs the non-terminal name **before** processing all its components, which violates the required output order. Additionally, there's a critical error in the **EqExp parsing** where the condition `"!==".contains(lexer.now().getName())` is incorrect - it should check for `"=="` or `"!="` tokens specifically.

This contains two errors:
1. The string `"!==".contains(...)` will match `"!"`, `"="`, or `"=="` individually, which is incorrect. It should only match `"=="` or `"!="`.
2. The premature output of `<EqExp>` inside the loop violates the required output order.

Similar issues exist in other expression parsing methods (AddExp, MulExp, RelExp, LAndExp, LOrExp) where the non-terminal name is printed inside the loop before all components are processed.
