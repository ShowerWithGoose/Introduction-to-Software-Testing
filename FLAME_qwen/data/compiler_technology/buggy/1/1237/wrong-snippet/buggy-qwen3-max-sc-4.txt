Based on the problem description and code review, the main issue appears to be in the error handling logic for missing semicolons and brackets. The error reporting mechanism incorrectly determines the line number for errors by reading an extra token after detecting the error condition.

The most problematic code is in the constant declaration parser where it handles missing semicolons:

Similar issues exist in other parsers like VarDeclParser, ConstDefParser, VarDefParser, etc. where the error line number determination logic is flawed. The code reads an extra token after detecting the missing symbol, then tries to backtrack to get the correct line number, but this approach is unreliable and doesn't correctly identify the line where the last valid non-terminal was parsed.
