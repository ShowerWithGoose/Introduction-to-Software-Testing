Based on the problem description and code review, the main issue appears to be in the error handling for missing semicolons and brackets. The error reporting logic incorrectly uses `unReadToken(2)` and then reads a token to get the line number, which may not correspond to the correct line of the last non-terminal before the missing symbol.

The error is that when a closing bracket or semicolon is missing, the code should report the line number of the last valid non-terminal token before the expected symbol, but the current implementation with `unReadToken(2)` followed by `readNextToken()` may not correctly identify this line number.
