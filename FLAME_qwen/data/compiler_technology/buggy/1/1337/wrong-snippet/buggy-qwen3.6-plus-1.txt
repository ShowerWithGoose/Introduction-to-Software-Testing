### Summary of Analysis
Although the test case input indicates a compilation environment failure (`g++: not found`), a static review of the provided source code reveals critical logical and semantic errors in the error reporting and file I/O modules:
1. **`util/Util.cpp`**: The `printErrorInfo` function completely ignores the `lineNum` parameter passed by the parser. Instead, it incorrectly outputs the line number of the previously consumed token (`Token::tokenList[Token::tokenIndex - 1].lineNum`). This violates the problem specification, which requires reporting the exact line number where the syntax error occurs.
2. **`error/Error.cpp`**: In `sortError()`, `std::getline` is used immediately after the `>>` extraction operator. The `>>` operator leaves the delimiter (space/newline) in the input stream, causing `getline` to capture a leading whitespace character in `temp.errorType`. This results in malformed error output (e.g., `1  a` instead of `1 a`).
3. **`lexer/Lexer.cpp`**: When a single `&` or `|` is encountered, the lexer reports error `a` but incorrectly adds a `"&&"` or `"||"` token to the token list. This will cause the parser to misinterpret the token stream and likely crash or produce incorrect syntax trees.
