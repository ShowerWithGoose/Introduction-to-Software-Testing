Based on the problem description and code review, the main issue appears to be in the `parseCompUnit()` method in Parser.java. The current implementation uses `peek(2) != TokenType.LPARENT` to determine Decl vs FuncDef, but this logic is flawed because it doesn't properly handle the grammar structure. According to the grammar, CompUnit should parse all Decl first (which start with 'const', 'int', or 'char'), then FuncDef (which start with 'void', 'int', or 'char' followed by an identifier and '('), and finally MainFuncDef.

The current peek condition will incorrectly stop parsing Decl when it encounters a FuncDef that starts with 'int' or 'char' (since those are valid BType tokens), causing FuncDef to be parsed as Decl.

    private CompUnitNode parseCompUnit() {
        //Decl
        ArrayList<DeclNode> declNodes = new ArrayList<>();
        while (peek(2) != TokenType.LPARENT) {
            declNodes.add(parseDecl());
        }

        //FuncDef
        ArrayList<FuncDefNode> funcDefNodes = new ArrayList<>();
        while (peek(1) != TokenType.MAINTK) {
            funcDefNodes.add(parseFuncDef());
        }

        //MainFuncDef
        MainFuncDefNode mainFuncDefNode = parseMainFuncDef();

        return new CompUnitNode(declNodes, funcDefNodes, mainFuncDefNode);
    }
