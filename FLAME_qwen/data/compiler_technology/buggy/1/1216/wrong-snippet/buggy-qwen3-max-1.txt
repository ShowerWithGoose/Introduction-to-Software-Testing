Based on the problem description and code structure, the issue appears to be in the error handling for missing semicolons. According to the requirements, when a semicolon is missing, the error line number should correspond to "the line where the last non-terminal before the semicolon is located". However, looking at the current implementation in multiple parser files, the error is being added immediately when the semicolon is not found, which would use the current token's line number (which might be the next token after where the semicolon should be).

The most likely location of the error is in the statement parsing logic where semicolons are expected. Specifically in `StmtParser.java` and `DeclParser.java` where `MISS_SEMICN` errors are handled.
