Based on the problem description and code review, the issue appears to be in the error handling for missing semicolons. According to the requirements, when a semicolon is missing, the error line number should correspond to "the line where the last non-terminal before the semicolon is located". However, in the current implementation, the error is being reported at the line of the previous token (via `getByOffset(-1)`), which may not correctly correspond to the last non-terminal's line.

The key issue is in the `TokenHandler.addError()` method which always uses the previous token's line number, but different error types need different line number calculations. Specifically for error type 'i' (missing semicolon), we need to track the line number of the last non-terminal component.

Looking at the parser code, the semicolon checks are done in multiple places (DeclParser, StmtParser, etc.), but they all use the same `handler.addError(ErrorType.MISS_SEMICN)` which reports the wrong line number.

    public void addError(ErrorType errorType) {
        errorList.add(new Error(errorType, getByOffset(-1).getLine()));
    }

The fundamental problem is that `addError()` always uses the previous token's line, but for missing semicolon errors, we need the line number of the last meaningful non-terminal construct, not just the previous token.
