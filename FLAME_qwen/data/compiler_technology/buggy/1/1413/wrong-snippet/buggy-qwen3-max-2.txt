Based on the problem description and code review, the main issue appears to be in the error handling for missing semicolons (error type 'i'). The error line number should correspond to "the line where the last non-terminal before the semicolon is located", but the current implementation often uses the line number of the assignment operator or other tokens instead of the actual last non-terminal.

Looking at multiple parser classes, I can see a pattern where the `errorLine` is set to the wrong token's line number. For example, in `StmtAssignParser`, the error line is set to the assignment operator's line rather than the last expression's line.

The most critical location is in the statement parsers where semicolon errors are handled:
