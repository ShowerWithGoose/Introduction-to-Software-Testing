### 1. Compilation Error: Naming Conflict in `PrimaryExp.java`
The classes `Number` and `Character` in the `frontend.nonTerminal` package share names with `java.lang.Number` and `java.lang.Character`. Since `PrimaryExp.java` does not explicitly import the custom classes, Java defaults to the `java.lang` versions. This causes a type mismatch when `Parser.java` tries to assign `frontend.nonTerminal.Number` and `frontend.nonTerminal.Character` objects to these fields.

### 2. Logical/Runtime Error: Incorrect Lookahead Index in `Parser.java`
The method `index2Token(int offset)` is defined as `lexer.getTokens().get(index + offset - 1)`. When `offset = -1`, it accesses `index - 2`. This is used repeatedly for error reporting (e.g., missing `;`, `]`), which will either throw an `IndexOutOfBoundsException` (if `index < 2`) or report the line number of the wrong token. Error reporting should use the line number of the last successfully parsed token or component, not a negative offset lookup.

### Recommended Fixes:
1. **In `PrimaryExp.java`**: Add explicit imports to resolve the ambiguity:
   ```java
   import frontend.nonTerminal.Number;
   import frontend.nonTerminal.Character;
   ```
   *(Alternatively, rename the custom classes to `MyNumber`/`MyCharacter` or `ConstNumber`/`ConstCharacter` to avoid `java.lang` conflicts entirely.)*
2. **In `Parser.java`**: Replace `index2Token(-1).getLine()` with a safer approach. For example, store the line number of the last matched token in a variable before parsing, or use `this.token.getLine()` (which points to the current mismatching token, but is safer than negative indexing). For missing `;` or `]`, the spec asks for the line of the *last non-terminal*, so tracking the line of the last successfully parsed component is ideal.
