### Summary
- **Problem Description**: Implement a lexical analyzer that reads `testfile.txt`, identifies tokens (keywords, identifiers, constants, operators, etc.) or lexical errors, and outputs results to `lexer.txt` (for correct programs) or `error.txt` (for erroneous programs). Error type `a` corresponds to standalone `&` or `|` symbols.
- **Test Case Input**: `/bin/sh: 1: javac: not found` (Indicates a shell/environment compilation failure in the test runner, but the core issue lies in the program's logic which would cause runtime crashes or incorrect tokenization).
- **Buggy/Expected Output**: Not provided, but the logical flaws will lead to `IndexOutOfBoundsException`, skipped characters, or incorrect line number tracking.
- **Error Location**: The `analyse` method in `frontend/Lexer.java` contains multiple critical logical flaws:
  1. **Index Management Bug**: The `for` loop automatically increments `i`, but the code also manually increments `i` inside `while` loops for multi-character tokens (strings, chars, comments). This causes `i` to advance too far, skipping characters or misaligning the parser.
  2. **Bounds Checking Bug**: In string constant parsing, `next = source.charAt(i + 1);` lacks a bounds check. If a string ends at the last character of the file, this throws an `IndexOutOfBoundsException`.
  3. **Line Number Tracking Bug**: Inside `//` and `/* */` comment parsing, newlines are consumed within the `while` loop, bypassing the `lineNum++` check at the top of the `for` loop. This results in incorrect line numbers for subsequent tokens/errors.
