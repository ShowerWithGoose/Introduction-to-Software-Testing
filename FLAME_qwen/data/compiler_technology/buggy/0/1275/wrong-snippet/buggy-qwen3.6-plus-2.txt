### Summary of Information & Error Identification

**Problem Overview:**
The task requires implementing a lexical analyzer that reads `testfile.txt`, identifies tokens (keywords, identifiers, constants, operators, delimiters), and outputs them to `lexer.txt`. If lexical errors occur (specifically error type `a` for standalone `&` or `|`), it should output the line number and error code to `error.txt`.

**Test Case & Output Context:**
The provided test input (`/bin/sh: 1: javac: not found`) indicates a system-level compilation failure in the test environment, but analyzing the source code reveals logical gaps that would cause incorrect behavior or missing functionality during actual execution.

**Identified Logical/Semantic Errors:**
1. **Missing `CharConst` (`'...'`) Parsing Logic:** The token table explicitly defines `CharConst` (`CHRCON`), and the `categoryCodes` map initializes it, but the `analyze` method completely lacks a branch to parse character constants enclosed in single quotes. This will cause character literals to be misclassified or trigger unrecognized symbol errors.
2. **Incorrect Line Number Tracking for Multi-line Strings:** When a string literal spans multiple lines, `lineNumber` is incremented inside the parsing loop. Consequently, the token is recorded with the *ending* line number instead of the *starting* line number, which violates standard lexical analysis conventions and may cause mismatched line reporting.
3. **Non-compliant Error Format for Unrecognized Symbols:** The fallback error handling `errors.add(lineNumber + ": Unrecognized symbol " + currentChar);` does not follow the required output format (`line number error category code`). It should output a standardized error code or be handled according to the assignment's error specification.

**Likely Error Location:**
The errors are concentrated in the `analyze` method of `frontend/Lexer.java`, specifically within the token parsing branches and error handling logic.
