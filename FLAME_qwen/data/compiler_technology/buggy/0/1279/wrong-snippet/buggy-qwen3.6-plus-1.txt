### Summary
Based on the problem description and a careful review of the source code, two logical/semantic errors have been identified:
1. **Token Category Code Mismatch**: In `Compiler.java`, the `tokenTypeToString` method maps `CHAR_CONST` to `"CHARCON"`. However, the provided specification table explicitly defines the category code for `CharConst` as `"CHRCON"`. This will cause incorrect output for character constants.
2. **Incorrect Lexeme Construction for Escape Characters**: In `frontend/Lexer.java`, the `recognizeCharConst` method incorrectly builds the token string for escape sequences. After calling `advance()` to consume the escape character, the code uses `(char) currentChar` in the concatenation. At this point, `currentChar` has already advanced past the escape character (and is likely pointing to the closing `'` or the next character), resulting in a malformed token lexeme. This violates the requirement to preserve the source string exactly as it appears.
