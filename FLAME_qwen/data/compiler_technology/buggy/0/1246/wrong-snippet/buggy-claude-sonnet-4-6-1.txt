## Summary

**Problem:** Lexical analyzer that tokenizes source code and detects error type `a` (single `&` or `|` instead of `&&` or `||`).

**Test Case Failure:** The compilation fails with `/bin/sh: 1: javac: not found` — meaning the Java source cannot be compiled at all. This is a **compilation error** in the Java source itself, not a logic error.

The most likely cause is a syntax/language-level issue. Looking at the code, `switch` expressions with `->` (arrow case labels) require **Java 14+**. If the grading environment only has an older JDK, this would cause a compile error.

The suspicious snippet is in `utils/IO.java`:

---

```utils/IO.java```
...
    public static String getPath(boolean output, IOType ioType) {
        // 读取输出路径
        if (output) {
            switch (ioType) {
                case CHECKER -> {
                    return Config.atLocal ? Config.localOutputErrorFilePath : Config.outputErrorFilePath;
                }
                // LEXER, PARSER
                default -> {
                    return Config.atLocal ? Config.localOutputLexerFilePath : Config.outputLexerFilePath;
                }
            }
        }
        // 读取输入路径
        else {
            return Config.atLocal ? Config.localInputFilePath : Config.inputFilePath;
        }
    }
...
```

The arrow-style `switch` statement (`case CHECKER -> { ... }`) is a modern Java feature (Java 14+). If the judge environment uses Java 8 or Java 11, this syntax is not supported and will cause `javac` to fail — or if `javac` itself is not installed/found, the submission packaging may also be wrong.

Additionally, there is a secondary logic concern in `frontend/Lexer.java` regarding the error-A detection path:

```frontend/Lexer.java```
```

The `else if (isErrorA(...))` branch only triggers when `pos < maxLength` (i.e., there's a next character). If `&` or `|` appears at the very last position of the file (`pos == maxLength`), the error would be silently missed.
