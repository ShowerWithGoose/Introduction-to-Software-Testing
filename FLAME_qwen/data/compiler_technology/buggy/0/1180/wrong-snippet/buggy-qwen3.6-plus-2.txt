### Summary of Information & Error Analysis

**Problem & Requirements:**
- The program must perform lexical analysis, outputting tokens as `category_code value` to `lexer.txt` for correct programs, or `line_number error_code` to `error.txt` for erroneous ones.
- Error type `a` corresponds to standalone `&` or `|`.
- Output format must strictly match `category_code value` (single space separated, no extra text).

**Test Case & Environment:**
- The provided test input `/bin/sh: 1: javac: not found` indicates a system-level compilation failure in the test environment. However, analyzing the source code reveals critical logical and formatting bugs that would cause incorrect output or runtime behavior even if compiled successfully.

**Identified Errors:**
1. **`token/Token.java` - Output Format Violation:** The `toString()` method appends `" at line " + line`, which directly violates the required output format (`word category code character/string form of the word`). The line number should not be included in the token output.
2. **`lexer/Lexer.java` - Pass-by-Value Index Bug:** In `processLine`, the index `p` and state `inName` are passed by value to `handleNonCommentChar`. Any manual `p++` inside the helper method (e.g., for parsing strings, comments, or multi-character operators like `&&`) is **local to that method** and does not update the caller's loop variable. Consequently, the outer loop unconditionally executes `p++`, causing characters to be processed multiple times, comments to fail to skip, and operators like `&&` to trigger false error `a` on the second `&`.
3. **`lexer/Lexer.java` - Loop Index Management:** The unconditional `p++` at the end of the `while` loop in `processLine` conflicts with internal index advancement, breaking the lexer's state machine.
