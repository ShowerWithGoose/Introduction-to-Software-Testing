## Summary

The error is a **compilation error**: `javac: not found`. This means the Java compiler isn't available in the environment, so the code never actually runs. However, looking at the source code itself for logical/semantic errors that would cause runtime problems, there are two issues to flag:

1. **`peekNextChar()` is off-by-one**: it checks `pos >= text.length()` but then accesses `text.charAt(pos + 1)`. When `pos` is at the last valid index (`text.length() - 1`), the guard passes but `pos + 1` is out of bounds.

2. **`handleSymbol()` reads `currentChar` after `advance()`** for the two-character operators, but the two-char check uses `tmp` (built from the *original* `currentChar` and `peekNextChar()`), while the second `sb.append(currentChar)` appends the *new* `currentChar` after advance — this is correct in logic but only works because `advance()` moves to the next char. The real bug is that `handleSymbol` calls `advance()` unconditionally for the first char, then checks `tmp` for two-char tokens. If the first char is `|` or `&` (single, error case), `tmp` would be `"|x"` or `"&x"` which won't match any two-char pattern — that part is fine. But the `peekNextChar` boundary issue is the structural bug.

The most likely error location:

```frontend/Lexer.java```
```
...
    private char peekNextChar() {
        if (pos >= text.length()) {
            return '\0';
        }
        return text.charAt(pos + 1);
    }
...
```

The guard should be `if (pos + 1 >= text.length())` — as written, when `pos` equals `text.length() - 1`, the check passes but `pos + 1` throws a `StringIndexOutOfBoundsException`.
