**Summary of the Problem & Test Case:**
The assignment requires a lexical analyzer that tokenizes source code according to a specific grammar and detects lexical errors. Specifically, standalone `&` or `|` symbols must be flagged as error type `a`. The program reads `testfile.txt`, outputs valid tokens to `lexer.txt`, and errors to `error.txt`. The provided test case input (`/bin/sh: 1: javac: not found`) reflects a system environment issue, but a static analysis of the provided source code reveals a critical logical flaw that will cause a runtime crash during lexical analysis.

**Error Location Analysis:**
The bug is located in `<lexer/Lexer.java>` within the `readNextToken()` method. The code repeatedly uses `source.charAt(curPos)` to peek at the next character when identifying two-character operators (`&&`, `||`, `!=`, `<=`, `>=`, `==`) and comment starters (`//`, `/*`). However, it **omits the boundary check `curPos < source.length()`** before calling `charAt()`. If the source file ends with a single `&`, `|`, or any other operator/comment starter, `curPos` will equal `source.length()`, causing a `StringIndexOutOfBoundsException`. This directly breaks the error type `a` detection logic for `&` and `|`, as well as other multi-character token parsing.
