## Summary

**Problem:** Lexical analysis program that tokenizes source code and detects errors (specifically illegal single `&` or `|` symbols).

**Test Case Failure:** The program fails to compile — `javac: not found` — meaning the build itself is broken. The error is a **compilation error**, not a runtime logic error.

**Root Cause:** `Compiler.java` imports `javax.swing.*`, which is unnecessary and may cause compilation issues in headless/restricted environments (e.g., a grading server without a full JDK that includes Swing). Beyond that, there are also logical bugs in the lexer worth flagging.

---

## Code Snippets Where Errors Are Located

**Compiler.java**
```
import frontend.ErrorRecord;
import frontend.Lexer;
import frontend.Token;

import javax.swing.*;
...
```

The `import javax.swing.*;` is unused and can cause a compilation failure in environments where the Swing library is not available (e.g., headless JDK builds or restricted grading environments).

---

Additionally, there are logical errors in the lexer that would cause incorrect behavior once compilation is fixed:

**frontend/Lexer.java**
```
        } else if (Character.isDigit(ch)) {
            if (ch == '0') {
                value = "0";
            } else {
                ch = reader.read();
                while (Character.isDigit(ch)) {
                    sb.append((char) ch);
                    ch = reader.read();
                }
                if (ch != -1) {
                    reader.unread(ch);
                }
                value = sb.toString();
            }
            token = new Token(value, LexType.INTCON, lineNum);
            return true;
```

When `ch != '0'`, the first digit (already appended to `sb`) is correct, but then `ch = reader.read()` is called immediately, skipping the loop entry with the already-read next character — this logic is actually fine structurally, but the `ch == '0'` branch short-circuits and never reads further digits, which is correct for `0` alone but inconsistent.

The more critical logical bug:

**frontend/Lexer.java**
```
                } else if (ch == '&' || ch == '|') {
                    reader.unread(nextChar);
                    addErrorRecord(ErrorType.ILLEGAL_SYMBOL, lineNum);
                    String right = (char) ch + String.valueOf((char) ch);
                    token = new Token(doubleChar.substring(0, 1), doubleCharOperators.get(right), lineNum);
                    return true;
                }
```

`doubleCharOperators.get(right)` where `right` is `"&&"` or `"||"` — but the token value is `doubleChar.substring(0, 1)` (just `&` or `|`). The `get(right)` call is fine, but if `ch` is `&` and `nextChar` is something other than `&`, `right` would still be `"&&"`, which is misleading. More importantly, `nextChar` was already unread, so the constructed `doubleChar` still contains the wrong second character — `doubleCharOperators.get(right)` uses a hardcoded reconstructed key, which is the correct workaround, but the token value being `doubleChar.substring(0, 1)` is correct.
