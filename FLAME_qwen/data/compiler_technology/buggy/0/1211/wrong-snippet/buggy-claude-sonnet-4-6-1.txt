## Summary

The test case fails with a **compilation error**: `/bin/sh: 1: javac: not found`. This means the Java compiler isn't available in the test environment, but looking at the source code itself for logical errors, the real issue is in `Compiler.java`.

The logic for deciding whether to write to `lexer.txt` or `error.txt` is **inverted**: `errorTable.isEmpty()` is checked *before* the lexer has finished running — wait, actually the lexer runs in the constructor, so that's fine. But the core bug is:

The `Lexer` constructor calls `analyse()` and populates both `tokens` and `errors`. Then in `Compiler.java`, the check `errorTable.isEmpty()` happens **after** construction, which is correct timing. However, the output logic writes `lexer.getTokens()` even when there are errors, and writes errors only when there are none — **the condition is backwards**.

Additionally, in `getType2Token`, when a single `&` or `|` is detected as an error, it still returns a token with the single character but uses the two-character `TokenType` (AND/OR), and crucially it **does not unread** the non-matching character, consuming it silently.
