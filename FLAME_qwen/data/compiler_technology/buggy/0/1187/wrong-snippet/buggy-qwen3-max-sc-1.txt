Based on the problem description and test case input indicating a compilation error due to missing `g++`, but more importantly focusing on the **lexical analysis logic** and the **error handling for type 'a'** (individual '&' or '|'), we examine the lexer's handling of these symbols.

The issue is in the error handling within the `lexer.cpp` file: when encountering a single `&` or `|`, the code correctly detects error type `a`, but **still returns a token of type `AND` or `OR`**, which is incorrect. According to the specification, **if there's a lexical error, the program should output only the error (to `error.txt`) and not produce any token output**. However, the current implementation both prints the error **and** continues processing as if it had read a valid `AND`/`OR` token, which contaminates the token stream.

Moreover, the main function checks `lexer->hasError` only **after** collecting all tokens, but during token collection, erroneous tokens are still being pushed into `token_vec`. This violates the requirement that **erroneous programs should only output error info, not tokens**.

But the **immediate lexical error misbehavior** (producing a token despite an error) occurs in this part of `lexer.cpp`.
