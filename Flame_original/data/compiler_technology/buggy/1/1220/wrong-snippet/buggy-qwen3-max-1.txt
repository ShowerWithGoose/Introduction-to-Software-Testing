Based on the problem description and code review, the issue appears to be in the error handling for missing semicolons in statement parsing. The test case shows a compilation error related to javac not being found, but the actual issue is likely in how the parser handles missing semicolons.

Looking at the StmtParser class, particularly in the default case and IDENFR case, there's inconsistent handling of missing semicolons. The error reporting uses the current token's line number rather than the line number of the last non-terminal before the expected semicolon, which violates the specification.
