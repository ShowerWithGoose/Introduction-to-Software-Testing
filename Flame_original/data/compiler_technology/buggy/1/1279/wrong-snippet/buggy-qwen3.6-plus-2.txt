### Summary of Errors
Based on the grammar rules and output requirements, the following logical and semantic errors are identified in the parser:
1. **Duplicate Output of Expression Non-terminals**: Methods like `parseLOrExp`, `parseLAndExp`, `parseEqExp`, `parseRelExp`, `parseAddExp`, and `parseMulExp` incorrectly place `outputGrammar()` calls inside `while` loops or before parsing completes. This causes the same syntax component name to be output multiple times. According to standard recursive descent parsing and the assignment requirements, each non-terminal should only be output **once at the end** of its parsing method.
2. **Duplicate Output of `<Stmt>`**: The main `parseStmt()` method outputs `<Stmt>` at the end, but all specific statement helper methods (`parseEmptyStmt`, `parseBlockStmt`, `parseIfStmt`, `parseReturnStmt`, `parsePrintfStmt`, `parseAssignmentStmt`, `parseArrayAssignmentStmt`, `parseFunctionCallStmt`, `parseExpressionStmt`, `parseGetStmt`, `parseForLoopStmt`, `parseBreakStmt`, `parseContinueStmt`) also output `<Stmt>`. This results in `<Stmt>` being printed twice for every statement.
3. **Incorrect Grammar Routing in `parsePrimaryExp`**: When `parsePrimaryExp` encounters an identifier followed by `(`, it calls `parseUnaryExp()`. According to the grammar `PrimaryExp → '(' Exp ')' | LVal | Number | Character`, an identifier should simply be parsed as an `LVal`. Function call syntax belongs to `UnaryExp`. Calling `parseUnaryExp()` here causes redundant parsing, incorrect token consumption, and broken output structure.
