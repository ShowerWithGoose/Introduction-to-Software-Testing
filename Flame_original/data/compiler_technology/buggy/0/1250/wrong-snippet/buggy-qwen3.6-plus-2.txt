**Summary**
The problem requires implementing a lexical analyzer that tokenizes source code into `lexer.txt` for valid programs or reports lexical errors to `error.txt` for invalid ones. The provided test case input (`/bin/sh: 1: javac: not found`) indicates an environment compilation failure, but a static analysis of the source code reveals several logical and semantic bugs in `frontend/Lexer.java`:
1. `getIntConst()`: The condition `if (sourceCode.charAt(charIndex - 1) != '0')` incorrectly halts digit consumption when an integer begins with `0`. This breaks tokenization for numbers like `01`, `0123`, etc., and contradicts the requirement to preserve numerical strings exactly.
2. `skipComment()`: The multi-line comment scanning loop accesses `sourceCode.charAt(charIndex + 1)` without verifying `charIndex + 1 < sourceCode.length()`. This will throw an `IndexOutOfBoundsException` if the source file ends immediately after a `*` inside a comment.
3. `getOperator()`: When a single `&` or `|` is encountered, the code incorrectly adds a valid `AND` or `OR` token to the token list before writing the error. According to the specification, these characters are strictly lexical errors and should not be emitted as valid operator tokens.
