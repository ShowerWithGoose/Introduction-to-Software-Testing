### Summary of Information
- **Problem Description**: Implement a lexical analyzer that outputs tokens to `lexer.txt` for correct programs, or error line numbers and error codes to `error.txt` for erroneous programs. The only lexical error defined is type `a` for standalone `&` or `|`. Numerical values must be output exactly as they appear in the source.
- **Test Case Input**: `/bin/sh: 1: javac: not found` (Indicates a compilation/environment failure in the test runner, but logical bugs exist in the provided source code).
- **Buggy/Expected Output**: Empty (Due to the compilation failure, the program did not execute to produce output).
- **Identified Errors**:
  1. **`Compiler.java`**: When outputting `INTCON` tokens, the code uses `token.getNumValue()` which converts the token to an integer. This violates the requirement to *"preserve the read string exactly as it appears"*. It should use `token.getTokenValue()`.
  2. **`TokenLexer.java`**: In the `isGreedyCh` block, `columnNum--` is executed unconditionally when `&` or `|` is not followed by a matching character. If the character appears at the end of a line, no lookahead character was consumed (`ch2` was never read). Decrementing `columnNum` in this case points the scanner back to the current character, causing an **infinite loop**. Additionally, after detecting an error, the code continues to add a `null`-type token to `tokenList`, which should be skipped.
