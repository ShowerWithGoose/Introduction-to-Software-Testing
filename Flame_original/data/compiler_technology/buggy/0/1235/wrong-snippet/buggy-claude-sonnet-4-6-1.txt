## Summary

**Problem:** Lexical analyzer in Java fails to compile with `javac: not found` — this is a build/environment error, not a logic error in the code itself.

**Test case input:** `/bin/sh: 1: javac: not found`

**Buggy output:** (empty)

**Expected output:** (empty — but the program should compile and run)

**Root cause:** The error `javac: not found` means the Java compiler is not installed or not on the PATH in the execution environment. However, looking at the source code itself, there is a structural issue that would cause a **compilation error** even if `javac` were available: the `Lexer.java` file uses `bufferedReader.mark(1)` and `bufferedReader.reset()`, but `BufferedReader.mark(int)` requires a read-ahead limit. With a limit of `1`, the mark may be invalidated before `reset()` is called in loops that read multiple characters — but more critically, the `BufferedReader` is never initialized with a sufficient buffer for `mark`/`reset` to work reliably.

But the most direct compilation error is in the `&`/`|` handling block — `bufferedReader.mark(1)` is called **after** `bufferedReader.read()` has already been called, meaning the mark is set on the wrong position, and `reset()` would not back up to the right character.

The likely code location of the logical/semantic error:

```frontend/Lexer.java
...

                character = bufferedReader.read();
                char nextCh = (char) character;
                currentToken.setLength(0);
                currentToken.append(ch);
                bufferedReader.mark(1);
                if((ch == '&' && nextCh == '&') || (ch == '|' && nextCh == '|')){
                    currentToken.append(nextCh);
                    String opToken = currentToken.toString();

                }else {
                    String opToken = currentToken.toString();
                    errors.add(new Error(opToken, "a", lineNum));
                    bufferedReader.reset();

...
```

The `bufferedReader.mark(1)` is placed **after** `bufferedReader.read()` has already consumed `nextCh`. So when the else-branch calls `bufferedReader.reset()`, it resets to the position **after** `nextCh`, not before it — meaning `nextCh` is silently dropped and never re-processed. The mark should be set **before** reading `nextCh`.
