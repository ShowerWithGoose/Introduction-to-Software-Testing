**Summary of the Issue:**
The problem description explicitly states: *"Ensure that the read string is preserved exactly as it appears for output purposes, especially for numerical values."* However, the current implementation in `Lexer.java`'s `parseInt()` method parses the sequence of digits into a primitive `int` value. This approach strips leading zeros (e.g., `007` becomes `7`) and risks integer overflow for large constants. Consequently, `Token.java`'s `toString()` method outputs this computed `intValue` instead of the original lexical string. Both locations must be modified to capture and output the exact string representation of the integer token.
