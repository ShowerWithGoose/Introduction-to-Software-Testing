**Summary of the Problem:**

This indicates that the lexer is incorrectly reporting the line number of the error. It seems to be detecting the error at line 10 instead of line 25.

Looking at the code, the `Lexer` class maintains a `line` variable. The `getToken` method checks for errors like standalone `&` or `|`. When such an error is found, it adds an `Error` object with the current `line` number.

The issue likely stems from how the `line` number is updated or how the error is detected.

Let's check if there is a bug in how `line` is updated.
```java
                line += token.getContent().split("\n").length - 1;
```
If `token.getContent()` is `" \n"`, split gives `[" ", ""]`? No, `" \n".split("\n")` results in `[" "]` if trailing empty strings are discarded?
Java's `String.split` discards trailing empty strings by default.
`" \n".split("\n")` -> `[" "]`. Length 1. `1 - 1 = 0`. Line does NOT increment.
`"\n".split("\n")` -> `[]`. Length 0. `0 - 1 = -1`. Line decrements?
`" \n ".split("\n")` -> `[" ", " "]`. Length 2. `2 - 1 = 1`. Line increments.

This is a very common bug with `split`.
If a line ends with `\n`, and the token is just `\n` (or whitespace ending in `\n`), `split("\n")` might not count the newline correctly if it's at the end of the string.

Example:
Token content: `"\n"`
`"\n".split("\n")` returns an empty array `[]` in Java?
Actually:
```java
System.out.println("\n".split("\n").length); // Prints 0
```
So `line += 0 - 1` => `line -= 1`.

Token content: `" \n"`
`" \n".split("\n")` returns `[" "]`. Length 1.
`line += 1 - 1` => `line += 0`.

Token content: `"\n "`
`"\n ".split("\n")` returns `["", " "]`. Length 2.
`line += 2 - 1` => `line += 1`.

So, if a `Blank` token consists of spaces followed by a newline (e.g., `"    \n"`), `split("\n")` produces `["    "]`. Length 1. Update is 0. The newline is effectively ignored for line counting!
If a `Blank` token is just a newline `"\n"`, update is -1.
If a `Blank` token is `"\n    "`, update is 1.

In typical code, indentation follows newlines. So `\n` is often followed by spaces/tabs.
If the source is:
`...\n    int...`
The lexer might match `\n    ` as a single `Blank` token?
Regex `\s+` is greedy. It will match `\n    ` as one token.
Content: `"\n    "`
Split: `["", "    "]`. Length 2. Update: +1. Correct.

But what if the newline is at the END of the file? Or before EOF?
Or what if there is a blank line?
`\n\n`
Matched as one token `\n\n`.
Split: `["", "", ""]`? No, trailing empties discarded. `["", ""]`. Length 2. Update +1.
But there are TWO newlines. So line should increase by 2.
`"\n\n".split("\n")` -> `["", ""]`. Length 2. `2-1=1`. Only increases by 1. Error!

So, using `split("\n").length - 1` is fundamentally flawed for counting newlines because it misses trailing empty strings which represent consecutive newlines or newlines at the end.

Correct way:
Count occurrences of `\n` in the string.
```java
int count = 0;
for (char c : token.getContent().toCharArray()) {
    if (c == '\n') count++;
}
line += count;
```
