    Let's look at the `next_token` function signature:
    `void next_token(std::istream &input, Token &token, int &lineno)`
    
    One common issue in student lexers is **not handling EOF correctly in the main loop or inside next_token**.
    
    Inside `next_token`:
    ```cpp
    char ch = input.get();
    // ...
    else if (ch == EOF) {
        token = Token(Token::EOF_TK, content, lineno);
    }
    ```
    `std::istream::get()` returns `int_type`, which can be `traits_type::eof()`. Assigning it to `char` is implementation-defined if `char` is signed/unsigned. If `char` is unsigned, `EOF` (-1) becomes 255. The check `ch == EOF` will **fail** because `255 != -1`.
    
    This is a classic C++ bug. `input.get()` should be assigned to an `int`, not a `char`, to correctly detect EOF.
    
    If `ch` is declared as `char`, and EOF is read:
    1. `ch` gets value 255 (0xFF).
    2. `ch == EOF` (which is -1) is false.
    3. It falls through to the final `else` block:
       ```cpp
       } else {
           token = Token(Token::ERR, content, lineno);
           ErrorReporter::report(lineno, "invalid character: " + content);
       }
       ```
    4. It reports an invalid character error instead of ending the token stream.
    5. The loop in `main` continues indefinitely or until other errors occur, never breaking on `EOF_TK`.
