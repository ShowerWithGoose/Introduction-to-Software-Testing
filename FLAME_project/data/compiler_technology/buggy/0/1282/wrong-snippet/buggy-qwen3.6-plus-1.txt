       `lexIdentKeyword`:
       Loop reads chars. Last `fgetc` reads the non-ident char (e.g., space).
       So `c` is the space.
       Method returns.
       If we execute `fgetc()` again, we skip the space.

       ```
       It does NOT call `fgetc`.
       So `c` is still `;`.
       Then `fgetc()` at end of switch is executed. `c` becomes next char.
       Then `fgetc()` at end of loop is executed. `c` becomes char after that.
       
       So for `;`, it skips one char.
       
       For `lexIdentKeyword`, `c` ends up being the delimiter.
       Then `fgetc()` at end of loop skips the delimiter.
       
       So yes, the universal `fgetc()` is incorrect because the branches already advance the reader to the next character to be processed (or the delimiter which needs to be processed in the next iteration).
       
       Actually, the standard pattern is:
       1. Read char.
       2. Process char (and read ahead as needed).
       3. Loop continues. The next iteration should process the character left in `c`.
       
       So the `fgetc()` at the end of the loop is WRONG. It should only be called if the branch did NOT consume the next character?
       
       No, look at `if (c == '\n')`.
       It calls `fgetc()`. So `c` is updated.
       Then universal `fgetc()` updates it again.
       
       Look at `lexIdentKeyword`.
       It leaves `c` as the first char AFTER the identifier.
       Then universal `fgetc()` skips that char.
       
       So the universal `fgetc()` is always skipping a character.
       
       **Correct Fix:** Remove the `fgetc()` at the end of the `while` loop in `getTokenStream`.
       
       But wait, if I remove it, what about branches that DON'T update `c`?

       - Char: updates `c` (to char after closing quote).

         - `/` comment: updates `c` (to char after comment/newline).
         - `;`: Does NOT update `c` internally?

           ```
           Then `fgetc()` at end of `lexSymbolComment` is executed.
           So `c` is updated.
