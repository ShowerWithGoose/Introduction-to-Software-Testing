4.  **Buggy Output**: The program produces no output (or likely crashes/produces incorrect token output without reporting the error). The `ErrorHandler` writes errors only if the list is not empty.
5.  **Error Location**:
    *   In `frontend/Lexer.java`, the `getSymbol()` method handles `&` and `|`.
    *   Current logic: It checks if the next character is the same (`&&` or `||`). If yes, it consumes both and creates a valid token. If **no** (i.e., it's a single `&` or `|`), it breaks out of the switch, increments `pos` by 1, and calls `Token.symbolToTokenType(symbol)`.
    *   In `frontend/token/Token.java`, `symbolToTokenType` returns `TokenType.INVALID` for unknown symbols like single `&` or `|`.
    *   The `Lexer` adds this `INVALID` token to the `tokenList`.
    *   Crucially, the `Lexer` **does not detect this as an error** and does not call `ErrorHandler.ERROR_HANDLER.addError()`. It just treats it as an invalid token in the stream.
    *   Since no error is added to the `ErrorHandler`, `error.txt` remains empty (or isn't written correctly depending on implementation details of empty lists), and the program likely fails because it shouldn't output tokens for an erroneous program, or simply doesn't report the error as required.
    *   The fix requires detecting when `symbolToTokenType` returns `INVALID` specifically for the cases of single `&` or `|` (or generally handling INVALID tokens as lexical errors of type 'a' if that's the only invalid symbol case defined, but strictly speaking, the prompt says "The symbols '&' and '|' appearing individually").
    *   Looking at `getSymbol`, when it encounters a single `&`, it creates a token with type `INVALID`. The lexer needs to recognize this specific situation as Error 'a' and report it, rather than just creating an invalid token.
