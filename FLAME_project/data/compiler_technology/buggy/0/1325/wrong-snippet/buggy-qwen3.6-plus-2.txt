Looking at the source code `num=num/10`, after parsing `num` (identifier), the lexer should encounter `/`.
In `Lexer.java`, the `getToken` method handles `/` by calling `parseCmt()`.
Let's trace `parseCmt()`:
```java
private void parseCmt() {
    column++; // Moves past '/'
    char ch = curLine.charAt(column); // Checks next char
    if (ch == '/') {
        lineCmt = true;
        column++;
    } else if (ch == '*') {
        blockCmt = true;
        column++;
    } else {
        System.err.println("Undefined suffix of /");
    }
}
```
If the character after `/` is not `/` or `*` (e.g., it is `1` in `num/10`), `parseCmt()` prints an error to stderr but **does not return a token**. It simply returns `void`. The control flow returns to `getToken()`. Since `parseCmt()` advanced `column` by 1 (past the `/`), the next iteration of the `while` loop in `getToken()` will look at the character `1`. It will then call `parseInt()`, parsing `10` as an integer. The `/` operator itself is never emitted as a `DIV` token.

The fix should be: if `/` is not followed by `/` or `*`, it should be treated as the `DIV` operator. Currently, it just consumes the `/` and does nothing else regarding token generation for that specific character, effectively swallowing the operator.
