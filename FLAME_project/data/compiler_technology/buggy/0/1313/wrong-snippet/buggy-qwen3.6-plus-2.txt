**Summary of the Problem:**
The user's program is a lexical analyzer. It failed a test case where the expected error line number is 25, but the program outputted 24.
The input code contains an error on line 25: `if ('a' <= str[i] & str[i] <= 'z')`. The single `&` is a lexical error (Error type 'a').
The discrepancy suggests that the lexer is counting lines incorrectly, likely treating line numbers as 0-indexed internally but failing to convert them to 1-indexed for the output, or there is an off-by-one error in how `curRow` is tracked relative to the input list.

Looking at `Lexer.java`:
1. `curRow` is initialized to 0.
2. `inputList` is populated by reading lines from the file. `ArrayList` indices are 0-based.
3. When an error is detected in `analyzeOperator`, it creates an `Error` with `curRow`.
4. `ErrorList.toString()` outputs `error.getErrorRow()`.
5. If `curRow` is 0 for the first line, then for line 25 (1-based), `curRow` would be 24 (0-based).
6. The output is `24 a`, expected `25 a`. This confirms the issue is that the row number stored/reported is 0-indexed, but the requirement specifies line numbers starting from 1.

The error location is in `frontend/lexicAnalysis/Lexer.java` where the `Error` object is created, specifically passing `curRow` which is 0-indexed. It should pass `curRow + 1`. Alternatively, the initialization or increment logic could be adjusted, but fixing it at the point of error creation or output is safer. However, looking at `Token` creation, it also uses `tokenRow`. If tokens are expected to have 1-based line numbers (common in compilers), then `curRow` should generally be `index + 1`.

Let's look at where `curRow` is used.
In `analyze()`: `curStr = inputList.get(curRow);` -> This implies `curRow` is an index into the list (0-based).
In `analyzeOperator()`: `Error error = new Error(ErrorType.a, curRow, curCol);`

The fix should be to report `curRow + 1` for the line number.
