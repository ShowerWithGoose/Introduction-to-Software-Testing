Look at the comment patterns:

private static final String SINGLE_LINE_COMMENT_FORMAT = "//.*?\n";

This uses a reluctant quantifier .*? followed by \n.

But what if the comment is at the **end of file**, with **no trailing newline**?

Then the pattern "//.*?\n" will **not match**, because there's no \n.

So the lexer will see "// zhe ..." and fail to skip it.

Instead, it will try to tokenize:

- First '/' → matches DIV_FORMAT → outputs DIV /
- Second '/' → matches DIV_FORMAT again → outputs DIV /
- Then "zhe" → matches IDENFR → outputs IDENFR zhe, etc.

That’s exactly what we see in the buggy output!

So the error is in the **SINGLE_LINE_COMMENT_FORMAT**: it requires a newline, but the input comment may be at EOF without one.

Thus, the fix should allow optional newline or EOF.

But the bug is in the pattern.
