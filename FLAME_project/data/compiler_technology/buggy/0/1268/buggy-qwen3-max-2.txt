// The real issue: In the test input, there is a line with a comment: `    printf("%d",a); // 输出 1`
// The lexer handles '//' by doing: lineNumber++; break;
// This causes the lexer to immediately move to the next line without finishing tokenizing the current line before the comment.
// However, in the provided test case, the tokens BEFORE the "//" are already processed correctly.
// So why is the output order wrong?

// Looking at expected vs buggy output:
// Buggy output has:
//     ...
//     PRINTFTK printf
//     LPARENT (
//     STRCON "%d"

//     STRCON "%d"

//     SEMICN ;
//     RBRACE }          <--- this '}' comes BEFORE 'return'
//     RETURNTK return
//     INTCON 0
//     SEMICN ;
//     RBRACE }

// This implies that the inner block's closing brace '}' is being parsed AFTER the 'return' statement, but it should be before.

// Wait — look at the source code structure:

//     {
//         int a = 1;
//         printf("%d",a); // 输出 1
//     }
//     return 0;

// So the '}' of the inner block should come before 'return'.

// In the buggy output, the '}' appears at the very end, after 'return 0;'.

// This suggests that the lexer did NOT tokenize the '}' after the comment line.

// Why? Because when the lexer encounters "//", it does:
//   lineNumber++;
//   break;

// And then the outer loop does: lineNumber++ again at the bottom.

// So for a line like:
//   printf("%d",a); // 输出 1
// The lexer processes tokens up to ';', then sees '/' followed by '/', so it does:
//   lineNumber++  --> now points to next line (the one with '}')
//   break --> exits inner while
// Then the outer loop does: lineNumber++ --> now points to line after '}'

// Therefore, the line containing '}' is NEVER processed!

// That’s the bug.
