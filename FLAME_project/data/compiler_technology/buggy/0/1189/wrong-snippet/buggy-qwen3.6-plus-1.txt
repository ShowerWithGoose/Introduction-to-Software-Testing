One common issue in lexical analyzers involving comments is how newlines are handled.
In the block handling `//` comments:
```cpp
    } else if (this->Char == '/') {
        getChar();
        if (this->Char == '/') {
            getChar();
            while (this->Char != '\n') {
                getChar();
            }
            lineNum++;
            return 0;
        } 
        // ...
```
If the file ends with a `//` comment that does *not* end with a newline character (i.e., EOF is reached immediately after the comment text), the `while (this->Char != '\n')` loop will continue calling `getChar()`. `getChar()` sets `this->Char` to `'\0'` when `curPos` exceeds `fileText.size()`. The loop condition `this->Char != '\n'` will remain true (since `'\0' != '\n'`), leading to an infinite loop or accessing out-of-bounds memory if `getChar` doesn't handle EOF gracefully by stopping increments. Although `getChar` checks `curPos < fileText.size()`, once it hits EOF, `Char` becomes `'\0'` and `curPos` stays at `fileText.size()`. The loop `while (this->Char != '\n')` will become an infinite loop because `Char` remains `'\0'`.

Similarly, for `/* */` comments:
```cpp
        } else if (this->Char == '*') {
            getChar();
            while (curPos < fileText.length()) {
                while (this->Char != '*') {
                    if (this->Char == '\n') lineNum++;
                    getChar();
                }
                while (this->Char == '*') {
                    getChar();
                }
                if (this->Char == '/') {
                    return 0;
                }
            }
            return 0;
        }
```
This logic seems mostly okay regarding bounds, but let's look at the single character handling.
