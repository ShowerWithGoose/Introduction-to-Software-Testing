    *   The `reportError` function writes to `errorFile`: `errorFile << line << " " << "a" << std::endl;`. This matches the required output format for errors (`line number error category code`).
    *   However, there is a logical issue in how `next()` handles the state after reporting an error for `&` or `|`. It sets `lexType = ILLEGAL` but does **not** consume the current character `ch` (it stays as `&` or `|`). The next call to `next()` will encounter the same character again, potentially leading to an infinite loop or repeated error reporting for the same character if the main loop doesn't handle `ILLEGAL` by forcing a progress step.
    *   More critically, in the `default` case of the switch statement and other error paths, `ch` is often consumed or handled. But for `&` and `|` errors, `ch` is not advanced.
    *   Let's look at the `main.cpp` loop:
        ```cpp
        while (lexer.getLexType() != LexType::FINISH) {
              lexer.next();
              if (lexer.getLexType() != LexType::FINISH) {
                  // ... output ...
              }
        }
        ```
        If `next()` encounters a single `&`, it reports the error, sets `lexType = ILLEGAL`, but leaves `ch` as `&`. The loop continues. `getLexType()` returns `ILLEGAL` (which is not `FINISH`). `next()` is called again. It enters the `else` block (since `&` is not space, digit, alpha, etc.), goes to `case '&'`, sees peek is not `&`, reports error again, sets `ILLEGAL`. Infinite loop.

    *   To fix this, when an error like single `&` or `|` is detected, the character must be consumed (advanced) so the lexer can proceed to the next character.

    *   Additionally, the `reportError` function prints to `cout` as well: `cout << errStr << endl;`. The problem description doesn't explicitly forbid this, but usually, lexical analyzers should only output to the specified files. However, the infinite loop is the critical logical error preventing correct termination or processing.
