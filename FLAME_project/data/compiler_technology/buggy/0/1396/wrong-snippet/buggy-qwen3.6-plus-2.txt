**Summary of the Problem:**
The program is a lexical analyzer. The test case contains a C-like source code with a single-line comment `// 输出 1`. The expected output shows that this comment should be ignored (not tokenized). However, the buggy output includes tokens for the comment content: `DIV /`, `DIV /`, and `INTCON 1`. This indicates that the lexer is not correctly handling or skipping single-line comments.

**Analysis:**
1.  In `Frontend/Lexer.java`, the logic for handling the `/` character (which starts comments) is commented out.
2.  Because the `/` handling is commented out, the lexer falls through to the `default` case in the `switch` statement.
3.  In the `default` case, it simply appends `/` to the string builder and reads the next character. It does this twice for `//`, creating two separate `/` tokens.
4.  Then it encounters `输出 1`. Since `输` and `出` are likely not recognized as standard alphanumeric or whitespace in the simple `isWhitespace` or identifier checks (depending on Java's `Character.isAlphabetic` behavior for Chinese characters, they might be treated as alphabetic, but the space breaks it), or more likely, the space separates them. The `1` is then picked up as an `INTCON`.
5.  The core issue is that the comment handling code in `Lexer.java` is disabled.

**Error Location:**
The error is in `Frontend/Lexer.java` where the comment handling logic is commented out.
