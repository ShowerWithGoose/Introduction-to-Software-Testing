    The `while(ch != '\n')` loop continues reading characters until a newline is found. However, if the file ends without a newline character after the comment (or if `read()` returns `-1` indicating End Of File), `ch` will become `(char)-1` (which is `\uffff`, not `\n`). The condition `ch != '\n'` remains true, and `fileReader.read()` continues to return `-1`. The loop never terminates, causing the Time Limit Exceeded error.
    The same issue exists in the multi-line comment block `/* ... */` where `while(character != -1)` is used, but inside that loop, if the closing `*/` is missing, it might also hang or behave unexpectedly, but the single-line comment bug is the most direct cause for EOF handling failure in many simple lexers if the last line is a comment without a trailing newline. Even if there is a newline, checking for `character == -1` inside the `//` loop is necessary to prevent hanging on malformed files or files ending abruptly.

    Therefore, the error is in the `while` loop handling single-line comments in `frontend/Lexer.java`.
