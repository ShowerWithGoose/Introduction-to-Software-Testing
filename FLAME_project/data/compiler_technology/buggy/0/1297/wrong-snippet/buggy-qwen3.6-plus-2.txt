**Summary of the Problem:**
The program is a lexical analyzer. It reads source code from `testfile.txt` and outputs tokens to `lexer.txt` or errors to `error.txt`.
The specific test case contains a single lexical error: a standalone `&` character on line 25 (`if ('a' <= str[i] & str[i] <= 'z')`).
The expected output is `25 a` (Line 25, Error type 'a').
The actual output is `24 a` (Line 24, Error type 'a').

This indicates an "Off-by-one" error in line number tracking. The lexer reports the error on the previous line.

**Analysis of the Error Location:**
In `Reader/LexicalAnalyzer.java`, the `getTokens()` method processes the input string character by character.
1.  It maintains a line counter `num`.
2.  When it encounters a newline `\n`, it increments `num` and continues.
3.  Crucially, when creating a `Token`, it passes `num` as the line number: `new Token(..., num)`.

Let's trace the logic around newlines:
-   Suppose we are at the end of Line 24. The character is `\n`.
-   The code executes:
    ```java
    if (s.charAt(i) == '\n') {
        num++; // num becomes 25 (assuming it was 24)
        i++;
        continue;
    }
    ```
-   Now `i` points to the first character of Line 25. `num` is 25.
-   The loop continues. If the next character is a space or tab, it is skipped (`i++`).
-   If the next character is part of a token (e.g., `&`), the token is created with `num` (which is 25).

However, look closely at the structure of the `if-else if` chain in `getTokens()`:

```java
            if (s.charAt(i) == ' ' || s.charAt(i) == '\t') {
                i++;
            }
            if (s.charAt(i) == '\n') { // <--- BUG HERE
                num++;
                i++;
                continue;
            }
```

If the current character is a space `' '`, the first `if` executes `i++`.
Then, the second `if` checks `s.charAt(i)` again.
If the line ends with spaces before the newline, e.g., `"   \n"`:
1.  `i` points to space. First `if` runs. `i` increments.
2.  Second `if` checks the next char. If it's still space, it does nothing (falls through).
3.  Loop repeats.
4.  Eventually `i` points to `\n`.
5.  First `if` (space/tab) is false.
6.  Second `if` (`\n`) is true. `num` increments. `i` increments. `continue`.

This logic seems generally correct for updating `num` *before* processing the content of the next line. So why is the line number wrong?

Let's look at how `num` is initialized and used.
`int num = 0;`
Lines start at 1.
When the file starts, `num` is 0.
If the first line has no leading newlines, tokens on Line 1 are created with `num=0`?
No, wait.
If `num` starts at 0, and we encounter the first `\n` at the end of Line 1, `num` becomes 1.
Then tokens on Line 2 are created with `num=1`.
This means tokens on Line 1 are created with `num=0`.
Tokens on Line 2 are created with `num=1`.
...
Tokens on Line N are created with `num = N - 1`.

So, if the error is on Line 25, the lexer likely assigns it `num = 24`.
This confirms the "Off-by-one" error due to initialization. `num` should start at 1, or the assignment should be `num + 1`.

Given the standard convention that line numbers start at 1, and `num` is incremented *after* seeing a newline (meaning `num` represents the count of newlines seen so far, which corresponds to the *previous* line index if we consider the first line as index 0 relative to newlines), the tokens on the first line are associated with 0 newlines seen.

If `num` is initialized to 0:
-   Line 1 tokens: `num` is 0.
-   End of Line 1: `\n` seen. `num` becomes 1.
-   Line 2 tokens: `num` is 1.
-   ...
-   Line 25 tokens: `num` is 24.

The expected output is Line 25. The actual output is 24.
Therefore, the error is that `num` is initialized to 0 instead of 1, or the line number assigned to the token is `num` instead of `num + 1` (if keeping `num` as newline count). The simplest fix is initializing `num` to 1.
