Based on the problem description and the provided code, the issue lies in the `Lexer.java` file. The program fails to produce any output (`No output file`) for a valid source program. This suggests that an exception is being thrown during the lexical analysis phase, causing the program to terminate before writing to `lexer.txt`.

Looking at the `scanString` method in `Lexer.java`:

```java
private void scanString() throws IOException {
    int chr = reader.read();
    assert chr == '"' : "Invariant broken";
    Pair<Integer, Integer> start = reader.location();
    StringBuilder str = new StringBuilder();
    while ((chr = reader.read()) != '"') {
        if (chr == '%') {
            chr = reader.read();
            if (chr != 'd') {
                errors.add(ErrorType.InvalidFormatString, location());
                continue;
            }
            str.append('%').append('d');
        } else if (chr == '\\') {
            chr = reader.read();
            if (chr != 'n') {
                reader.unread(chr);
                errors.add(ErrorType.InvalidFormatString, location());
                continue;
            }
            str.append('\n');
        } else if (chr != (char) 32 && chr != (char) 33 && (chr < (char) 40 || chr > (char) 126)) {
            errors.add(ErrorType.InvalidFormatString, location());
        } else {
            str.append((char) chr);
        }
    }
    addToken(TokenType.FString, '"' + str.toString() + '"', start);
}
```

The test case input contains string literals like `"hello\a world\t"` and `"%d\n"`.
1. `"hello\a world\t"`: The lexer encounters `\a`. It reads `\`, then reads `a`. Since `a` is not `n`, it adds an error `InvalidFormatString` and continues. Then it encounters `\t`. It reads `\`, then reads `t`. Since `t` is not `n`, it adds an error and continues. This logic seems flawed for general escape sequences if only `\n` is supported, but more importantly, let's look at the character validation.
2. The condition `chr != (char) 32 && chr != (char) 33 && (chr < (char) 40 || chr > (char) 126)` checks for printable ASCII characters (space, !, and range 40-126). However, standard strings often contain other characters or escape sequences.
3. Crucially, the `scanString` method does not handle EOF correctly inside the loop. If the string is unterminated, `reader.read()` returns `-1`. The loop condition `while ((chr = reader.read()) != '"')` will be true if `chr` is `-1` (since `-1 != '"'`). Inside the loop, if `chr` is `-1`, it might cause issues or infinite loops depending on subsequent reads. However, the test case has valid strings.

The character `'` (single quote) is **not** in the `symbols` map. Therefore, the assertion `assert symbols.containsKey((char) chr)` will fail when the lexer encounters a single quote, throwing an `AssertionError`. This causes the program to crash immediately without producing any output file.

Additionally, even if the assertion didn't crash, the lexer lacks support for:
1.  Scanning character constants (`CHRCON`).
2.  Recognizing `char` keyword (`CHARTK`).
3.  Recognizing `getchar` keyword (`GETCHARTK`).

The primary cause of "No output file" is the crash due to the unhandled single quote character.
