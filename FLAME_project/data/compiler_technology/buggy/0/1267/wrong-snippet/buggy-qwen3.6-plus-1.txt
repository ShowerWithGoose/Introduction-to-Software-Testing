    **Is there a bug in `lexer_getWord` regarding `seekg`?**
    
    ```cpp
    if (!isNum(ch)) file.seekg(-1,ios::cur);
    ```
    
    If `ch` is EOF? `isNum(EOF)`?
    `isNum` takes `char`. `EOF` is usually -1.
    If `char` is signed, -1 is not a digit.
    `seekg(-1)` on EOF state?
    
    If `file.get(ch)` hits EOF, `ch` is unchanged.
    If `ch` was initialized to `temp` (a digit), `isNum(ch)` is true.
    Loop continues?
    `while (!file.eof())`
    `file.get(ch)` -> EOF.
    `isNum(ch)` -> True (if `ch` retained old value).
    `singleWord.push_back(ch)` -> Pushes old digit again?
    
    **This is a bug!**
    
    ```cpp
    else if (isNum(temp)) {
        char ch = temp;
        singleWord.push_back(temp);
        while (!file.eof()) {
            file.get(ch);
            if (isNum(ch)) {
                singleWord.push_back(ch);
            }
            else break;
        }
        if (!isNum(ch)) file.seekg(-1,ios::cur);
        // ...
    }
    ```
    
    If `file.get(ch)` fails (EOF), `ch` retains its previous value (the last valid digit).
    `isNum(ch)` is true.
    It pushes the digit again into `singleWord`.
    Then `while (!file.eof())` checks EOF. It is now EOF. Loop terminates.
    `if (!isNum(ch))` -> False.
    So it does NOT seek back.
    
    Result: The last digit is duplicated in the token string.
    Example: `12` at end of file.
    1. `temp` = '1'.
    2. `ch` = '1'. `singleWord` = "1".
    3. Loop. `file.get(ch)` reads '2'. `ch`='2'.
    4. `isNum('2')` true. `singleWord` = "12".
    5. Loop. `file.get(ch)` hits EOF. `ch` remains '2'.
    6. `isNum('2')` true. `singleWord` = "122".
    7. Loop condition `!file.eof()` is false. Terminate.
    8. `!isNum('2')` is false. No seek.
    9. Returns "122".
    
    This is definitely a bug. It affects `INTCON`, `IDENFR` (similar logic), and potentially others.

    Similarly for `IDENFR`:
    ```cpp
    else if (isAZ(temp) || temp == '_') {
        char ch = temp;
        singleWord.push_back(temp);
        while (!file.eof()) {
            file.get(ch);
            if (isNum(ch) || isAZ(ch) || ch == '_') {
                singleWord.push_back(ch);
            }
            else break;
        }
        if (!(isNum(ch) || isAZ(ch) || ch == '_')) file.seekg(-1,ios::cur);
        // ...
    }
    ```
    Same bug. If EOF occurs, `ch` retains last valid char. Condition is true. Duplicates last char.

    This bug causes incorrect token strings.
