In `Lexer.cpp`, `skipMultiLineComment` has a bug.
```cpp
void skipMultiLineComment(FILE *fp) {
    char ch;
    int nestedCount = 1;

    while ((ch = getChar(fp)) != EOF) {
        if (ch == '*') {
            ch = getChar(fp);
            if (ch == '/') {
                nestedCount--;
                if (nestedCount == 0) {
                    break;
                }
            }
        } else if (ch == '/') {
            ch = getChar(fp);
            if (ch == '*') {
                nestedCount++;
            } else {
                ungetc(ch, fp);
            }
        }
        if (ch == '\n') { // <--- BUG: ch might have been overwritten by the inner getChar calls above.
            line_count++;
        }
    }
}
```
If `ch` is `*`, it reads the next char into `ch`. If that next char is NOT `/`, the loop continues. The check `if (ch == '\n')` uses the *second* character read. If the sequence is `*\n`, it reads `\n` into `ch` inside the `if(ch=='*')` block. Then it checks `if (ch == '\n')` which is true. This seems okay for counting newlines inside comments.
But if the sequence is `/* comment */`, and inside the comment there is a newline.
Example: `/* \n */`.
1. `getChar` returns `/`. `ch` is `/`.
2. `else if (ch == '/')`: `getChar` returns `*`. `ch` becomes `*`. `nestedCount` becomes 2.
3. Loop continues. `getChar` returns `\n`. `ch` is `\n`.
4. `if (ch == '*')` false. `else if (ch == '/')` false.
5. `if (ch == '\n')` true. `line_count++`. Correct.
6. Loop continues. `getChar` returns `*`. `ch` is `*`.
7. `if (ch == '*')`: `getChar` returns `/`. `ch` becomes `/`. `nestedCount` becomes 1.
8. `if (ch == '\n')` false.
9. Loop continues. `getChar` returns EOF? Or next char.

Wait, standard C comments `/* ... */` do not nest in C/C++. The grammar doesn't specify nested comments, but the code implements nesting (`nestedCount`). Standard C compilers do not support nested comments. If the test cases assume standard C behavior, this might be an issue, but usually, simple comment skipping is fine.

If `tokenIndex` is not correctly pointing to the "last non-terminal" or if `tokenIndex - 1` is out of bounds (e.g., at the very beginning of parsing, though unlikely for syntax errors), this will produce wrong line numbers.
Specifically, if `getNextToken()` has advanced `tokenIndex` to the token *after* the one causing the error context, `tokenIndex - 1` is the current token.
In `ConstDecl`, if `;` is missing, `token` is the current token (not `;`). `tokenIndex` points to this `token`. `tokenIndex - 1` points to the previous token.
The requirement says: "The error line number corresponds to the line where the last non-terminal before the semicolon is located."
If the input is `int a = 1`, `token` is `1` (INTCON). `tokenIndex` points to `1`. `tokenIndex-1` points to `=`. The last non-terminal before the missing `;` is likely the `Exp` or `ConstInitVal` which ends with `1`. So `token.lineNum` (passed as arg) is correct. But the function uses `tokenList[tokenIndex-1]`, which is `=`. This might be wrong if `=` is considered a terminal separator and not part of the non-terminal's "location". Usually, the location of the error is the last token of the construct.
So, `Util::printErrorInfo` should use the passed `lineNum`.
