The bug is that when a single '&' is encountered, the lexer incorrectly adds a token "AND" with string "&&" instead of not adding any valid token or adding an invalid token. This causes the parser to see "&&" instead of "&", so it doesn't recognize there's an error.

The correct behavior should be to either:
1. Not add any token for the invalid single '&', or 
2. Add a token that represents the single '&' character

But instead, it's adding a fake "&&" token, which makes the parser think it's valid.

However, looking at the expected behavior, the lexer should detect the single '&' and report error 'a', but the token should still be processed in some way. The real issue might be elsewhere.

Looking at the Lexer.toString() and Lexer.printWords() methods, I notice that when flag is true (errors detected), it only outputs the wrongs from the lexer's own wrongs map. However, in the main Compiler class, it calls lexer.recognize() and then proceeds to do syntax analysis with CompUnit regardless of whether the lexer found errors.

This means that even if the lexer detects an 'a' error, the syntax analyzer still runs and may override or miss the lexer's error reporting.

But wait, looking at the WrongFile singleton pattern, both lexer and parser use the same WrongFile instance. However, the Lexer class has its own `wrongs` HashMap and `flag`, separate from the WrongFile singleton used by the parser.

This is the key issue! The Lexer class maintains its own error tracking (`wrongs` HashMap and `flag` boolean), but the main Compiler class only checks `WrongFile.getInstance().getFlag()` to decide whether to output parser results or error results. Since the lexer's errors are stored in its local `wrongs` map and not in the shared `WrongFile` singleton, they are never output.

The lexer should be using `WrongFile.getInstance().addWrong(wrong)` instead of maintaining its own `wrongs` map.
