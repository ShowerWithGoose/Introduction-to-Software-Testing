The `Lexer` has a significant bug in handling comments and potentially multi-line logic, but for the specific input:
```c
int main() {
    printf("21373119");
    int a = 0;
    {
        int a = 1;
        printf("%d",a); // 输出 1
    }
    return 0;
}
```
The lexer seems to handle single-line comments (`//`) by breaking the line loop. However, there is a logic error in `Lexer.tokenize()` regarding how it processes lines and the `flag` variable for multi-line comments. Specifically, `flag` is initialized to 0 outside the loop over lines, but if a multi-line comment starts and doesn't end on the same line, `flag` becomes 1. On the next line, if `flag` is 1, it looks for `*/`. If not found, it `break`s the inner `while` loop (processing characters of that line), but then continues to the **next entry** in `allLines`. This part might be okay for simple cases, but let's look closer at `Parser.java`.

Wait, `new FileWriter(parserOutputFileName)` creates the file. If `FileReader` throws an exception, the `try` block jumps to `catch`. The writers are closed in the `finally`? No, there is no `finally`. They are closed at the end of the `try` block. If an exception occurs at `new FileReader(...)`, the code jumps to `catch`, skipping the closing of writers. Let's look at another potential crash point. In `Parser.java`, `checkNext()` returns `null` if there is no next token.

In `parseCompUnit`:

If `checkNext()` returns `null`, calling `.getType()` on it will throw a `NullPointerException`. This would crash the parser before any output is generated (or before the output stream is flushed/closed properly if the crash happens inside the try block).

For the input `int main() ...`, the first token is `int`. `checkNext()` is `main`. This is safe.

However, consider the end of the file. After parsing `MainFuncDef`, `currentToken` should be `null` if all tokens are consumed.
In `parseMain`:
```java
main.addChild(parseBlock());
return main;
```
`parseBlock` consumes `{ ... }`. If the block is parsed correctly, `currentToken` advances.
If `parseBlock` finishes, `currentToken` is the token after `}`. In the example, `}` is the last token. So `currentToken` becomes `null`.
Then `parseCompUnit` loop condition `while (currentToken != null)` fails. Loop ends.

Let's look at `Compiler.java` file reading again.
```java
BufferedReader reader = new BufferedReader(new FileReader("C:\\Users\\86136\\Desktop\\hw2_Lexer\\hw2_Lexer\\testfile.txt"));
```
This is the most likely culprit for "No output file" in a generic testing environment. The file path is absolute and specific to the student's machine. The grader likely places `testfile.txt` in the current working directory. The code fails to find the file at the hardcoded path, throws `FileNotFoundException`, prints to stderr, and exits. The `parser.txt` and `error.txt` files might be created by `new FileWriter` (which truncates/creates them) but since the exception happens before any writing or closing, they remain empty or the OS might not even finalize their creation depending on buffering/OS behavior, or simply the grader sees empty files.

Actually, `new FileWriter(parserOutputFileName)` is executed **before** `new FileReader(...)`.
```java
BufferedWriter parserWriter = new BufferedWriter(new FileWriter(parserOutputFileName));
BufferedWriter errorWriter = new BufferedWriter(new FileWriter(errorOutputFileName));
// ...
BufferedReader reader = new BufferedReader(new FileReader("...")); // Exception here
```
So `parser.txt` and `error.txt` ARE created (empty). Then the exception occurs. The `catch` block executes. The writers are NOT closed. The buffers are NOT flushed. The files remain empty. The grader expects content in `parser.txt`. It finds an empty file. This matches "No output file" (interpreted as empty/missing content).

The fix is to use the relative path `"testfile.txt"` instead of the absolute path.
