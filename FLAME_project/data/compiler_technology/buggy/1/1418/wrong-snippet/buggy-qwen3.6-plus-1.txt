Based on the problem description and the test case, the program fails to output any error file (`error.txt`) when it should detect three "missing semicolon" errors (type `i`).

The root cause lies in how the parser handles errors and the control flow in `Compiler.java`.
1.  **Error Detection Logic**: The parsers (e.g., `VarDeclParser`, `StmtParser`) correctly identify missing semicolons and call `ErrorSave.error()`. This sets `GlobalData.Errorflag = true` and `GlobalData.Parserflag = false`.
2.  **Control Flow in `Compiler.java`**:
    *   The code checks `if(GlobalData.Parserflag)` to decide whether to write `parser.txt`. Since an error occurred, `Parserflag` is false, so `parser.txt` is not written. This is correct behavior for erroneous input (usually we don't output parse tree for invalid code).
    *   However, the code then checks `if(GlobalData.Errorflag)` to write `error.txt`.
    *   Looking at `Compiler.java`:
        ```java
        //ParserµłÉÕŖ¤ń╗ōµ×£ĶŠōÕć║Õł░ parser.txt

            bw = new BufferedWriter(new FileWriter(ParserPrintFilePath));

            throw new RuntimeException(e);
        }

            // ... writes to bw ...
            bw.close();
        }

        //ĶŠōÕć║ķöÖĶ»»ń╗ōµ×£Õł░ error.txt
        if(GlobalData.Errorflag) {
            ErrorPrint.run(GlobalData.Errors);
        }
        ```
    *   If `GlobalData.Parserflag` is false, the `bw` (BufferedWriter for `parser.txt`) is opened but **never closed** because the `bw.close()` is inside the `if` block. While this might not directly prevent `error.txt` from being created, it's bad practice.
    *   More importantly, let's look at `ErrorSave.java`. It sets `GlobalData.Errorflag = true`.
    *   Let's look at `Compiler.java` again. Is there an exception thrown?
