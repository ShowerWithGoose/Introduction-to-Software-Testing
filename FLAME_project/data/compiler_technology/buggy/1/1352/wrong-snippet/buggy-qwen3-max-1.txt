Based on the problem description and test case analysis:

1. **Expected vs Actual Output**: 
   - Expected: `16 j`, `18 a`, `19 j` (sorted by line number)
   - Actual: `18 a`, `16 j`, `18 a`, `19 j` (duplicate error, wrong order)

2. **Key Observations**:
   - Error 'a' (illegal symbol) appears twice for line 18 in actual output
   - Errors are not being properly deduplicated or sorted
   - The duplicate error suggests multiple error reports for the same issue

3. **Root Cause Analysis**:
   - The `&` symbol in `if (f2() & f1())` should trigger **one** error 'a'
   - The lexer already handles single `&` as illegal symbol (in `Lexer.java`)
   - However, the parser's `LOrExp`/`LAndExp` might be **re-reporting** the same error during syntax analysis
   - Specifically, when parsing `Cond → LOrExp`, the invalid `&` token causes multiple error reports

4. **Critical Code Location**:
   - The parser's expression handling (`LOrExp.java`, `LAndExp.java`) doesn't properly skip invalid tokens after reporting errors
   - This causes the same error to be reported multiple times during recursive descent
