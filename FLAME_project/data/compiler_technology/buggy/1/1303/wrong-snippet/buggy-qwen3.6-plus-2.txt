Based on the problem description and the comparison between the buggy output (empty) and the expected output, the program fails to produce any output for a correct source program. The `Compiler.java` main method calls `lexer.printTokens`, which only writes to `lexer.txt` if there are no lexical errors. However, the `Parser` constructor is called immediately after, which performs the syntax analysis and should write to `parser.txt`.

The issue lies in the `Parser.java` file. Specifically, the `parse_Number` and `parse_Character` methods are incorrectly implemented. They attempt to match `INTTK` ("int") and `CHARTK` ("char") respectively, instead of matching `INTCON` (integer constant) and `CHRCON` (character constant). This causes the parsing of expressions involving numbers (like `0` and `1` in the test case) to fail. When parsing fails, the parser might return null or enter an error state, but more importantly, looking at the `parse_Stmt` logic, if `parse_Exp` fails (because `parse_AddExp` -> `parse_MulExp` -> `parse_UnaryExp` -> `parse_PrimaryExp` -> `parse_Number` fails), the statement parsing fails.

However, the most critical error causing the **empty output** is likely that the `print()` method in `Parser` is called in the constructor, but if the parsing logic encounters a fatal error or returns null for the `CompUnit`, it might not print correctly, or the exception handling suppresses output. But looking closer at the `parse_Number` and `parse_Character` functions:

```java
    public Component parse_Number(){
        ArrayList<Component> components=new ArrayList<Component>();
        IntWrapper start= new IntWrapper(index);
        if(match(TokenType.INTTK,components)==null){ // ERROR: Should be INTCON
            jump(start);
            return null;
        }
        Component number =new Component(ComponentType.Number,components);
        return number;
    }
    public Component parse_Character(){
        ArrayList<Component> components=new ArrayList<Component>();
        IntWrapper start= new IntWrapper(index);
        if(match(TokenType.CHARTK,components)==null){ // ERROR: Should be CHRCON
            jump(start);
            return null;
        }
        Component character =new Component(ComponentType.Character,components);
        return character;
    }
```

Because `parse_Number` expects the keyword `int` instead of an integer literal, parsing `int a = 0;` will fail when trying to parse the `0` as an `InitVal` -> `Exp` -> ... -> `Number`. This causes the `parse_VarDecl` or `parse_Stmt` to fail. If the top-level `parse_CompUnit` fails or returns null due to these cascading failures, the `compUnit` might be null or incomplete.

In `Parser.java`:
```java
    public Parser(ArrayList<Token> tokens, ArrayList<Wrong> wrongs){
```

If `parse_CompUnit` returns `null` (which it does if `parse_MainFuncDef` fails, which it will if internal statements fail to parse correctly depending on how backtracking/jumping is handled in `parse_Stmt`), then `root` in `print()` becomes `null`.

```java

        OutputStream out = new BufferedOutputStream(new FileOutputStream("parser.txt"));
        PrintWriter writer = new PrintWriter(out);
        Component root=new Component();
        root=this.compUnit; // If compUnit is null, root is null
        postOrderTraversal(root,writer); // postOrderTraversal checks for null component

    }

        // ...
    }
```

Thus, the root cause is the incorrect token types in `parse_Number` and `parse_Character`.
