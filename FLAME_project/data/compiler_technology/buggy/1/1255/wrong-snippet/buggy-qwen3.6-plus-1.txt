The logic in `parseCompUnit` attempts to lookahead to distinguish between `Decl`, `FuncDef`, and `MainFuncDef`. However, the lookahead mechanism combined with `getBack` and the subsequent `parseDecl` call is flawed.

1.  `getNextToken()` is called twice, advancing `pos`.
2.  `getBack(2)` resets `pos` to the state before the two `getNextToken` calls? No, `getBack` subtracts from `pos`. If `pos` was `P`, after two `getNextToken`s it is `P+2`. `getBack(2)` makes it `P`. So `nowToken` is the token at `P`.
3.  The loop continues. `parseDecl()` is called. `parseDecl` expects to start parsing from `nowToken`.
4.  Inside `parseDecl`, `parseVarDecl` is called. It parses `BType` (`int`), then `VarDef` (`a1[10]`), then `;`.
5.  After `parseDecl` returns, the `while(true)` loop repeats.
6.  The issue is likely in how `parseDecl` consumes tokens versus how the lookahead in `parseCompUnit` manages the position.

The condition in the FuncDef loop only checks if the *current* token is `MAINTK`. It does not look ahead to see if the *next* token is `MAINTK` (which would happen if we were looking at `int main`). Or rather, it should check if the current construct is `int main`.

Since `FuncDef` starts with `FuncType` (`int`/`void`/`char`), and `MainFuncDef` starts with `int main`, the parser needs to distinguish `int ident` (FuncDef) from `int main` (MainFuncDef).

The current code parses `int main` as a `FuncDef` because it doesn't peek far enough or check the identifier name in the FuncDef loop.
