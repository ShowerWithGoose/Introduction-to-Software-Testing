Based on the test case and error output discrepancy, the issue is in how the parser determines the line number for missing semicolon errors (error code 'i'). The expected output shows errors on lines 2, 9, and 13, but the buggy output shows lines 2, 11, and 14.

Looking at the test input:
- Line 2: `printf("Hello SysY!\n")` (missing semicolon)
- Line 9: `int a` (missing semicolon) 
- Line 13: `return 0` (missing semicolon)

The problem appears to be in the `parseStmt()` method where it handles different statement types and reports missing semicolons. The error line number should correspond to the line where the last non-terminal before the semicolon is located, but the current implementation uses `iterator.peekPreviousToken().getLine()` which may not point to the correct line.
