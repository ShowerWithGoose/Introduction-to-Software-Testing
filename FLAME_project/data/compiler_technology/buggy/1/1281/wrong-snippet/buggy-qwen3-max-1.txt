Based on the test case and error output discrepancy, the issue is in how the parser determines the line number for missing semicolon errors (error code 'i'). The expected output shows errors on lines 2, 9, and 13, but the actual output shows lines 2, 11, and 14. This suggests the parser is using the wrong token's line number when reporting the error.

Looking at the test input:
- Line 2: `printf("Hello SysY!\n")` (missing semicolon)
- Line 9: `int a` (missing semicolon) 
- Line 13: `return 0` (missing semicolon)

The parser should report the line number of the last token before the expected semicolon, but it appears to be using the line number of a subsequent token or incorrect token.

The error occurs in multiple statement parsing methods where semicolon errors are reported. The key issue is in the `parseStmt()` method and related statement parsing methods where `iterator.peekPreviousToken().getLine()` is used to determine the error line number.
