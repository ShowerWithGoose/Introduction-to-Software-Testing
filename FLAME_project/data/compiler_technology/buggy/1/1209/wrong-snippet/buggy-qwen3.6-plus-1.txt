The error lies in the `expect` method of the `ASTNode` class. When a syntax error (missing semicolon, right parenthesis, or right bracket) is detected, the code adds the error to a local `errors` list but **does not advance the token index**. This causes the parser to get stuck on the same unexpected token, leading to infinite loops or repeated error reporting for the same location as different non-terminals attempt to parse and fail on the same token.

Additionally, the `addError` mechanism in `ASTNode` relies on `addErrors()` being called in `toString()`. However, if the parser gets stuck or throws an exception before reaching `toString()`, or if the error collection logic in `CompUnit` or other nodes doesn't properly propagate these local errors to the global `ErrorCollector` in a way that prevents duplicates or handles the "stuck" state, we get multiple errors.

A simpler fix for the specific duplication issue caused by `expect` not advancing is that `expect` *should* probably advance or handle the error such that subsequent parses don't re-report it if they are just retries. But the root cause is the retry logic in `StmtNode` combined with `expect` adding errors to a list that isn't cleared or checked for duplicates globally before output.

However, the most direct "bug" in the provided code structure that leads to *incorrect* behavior (duplicates) is that `ASTNode.expect` adds to `this.errors`, and `addErrors()` pushes them to the global collector. When `StmtNode` resets the index and reparses, new AST nodes are created. These new nodes have their own `errors` lists. They encounter the same syntax error, add it to their local list, and then `toString()` pushes it to the global collector again.

To fix this, one approach is to make `expect` advance past the error (panic mode) or to ensure that the global `ErrorCollector` handles duplicates (though line+type might be valid duplicates in some cases, here it's clearly unwanted).
