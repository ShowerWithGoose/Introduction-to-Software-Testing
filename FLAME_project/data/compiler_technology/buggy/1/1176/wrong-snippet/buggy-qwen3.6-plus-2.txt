The `CompUnit` grammar is `{Decl} {FuncDef} MainFuncDef`.
The current implementation uses a `while` loop that checks `isConst() || isFuncType()`.
1. When the parser encounters `int main()`, `isFuncType()` returns true (because `int` is a func type).
2. Inside the `else if (isFuncType())` block, it checks if the next token is an identifier and the one after is `(`. For `main`, this is true.
3. However, it then calls `FuncDef(root)`. `FuncDef` expects a return type, an identifier, parameters, and a block. It will consume `int`, `main`, `(`, `)`, and the block.
4. After `FuncDef` returns, the loop continues. The next token might be EOF or something else.
5. Crucially, the specific check for `MainFuncDef`:

   This check is inside the `while` loop but **after** the `if/else if` block that handles `Decl` and `FuncDef`. If `FuncDef` was called (consuming `int main...`), `curLex` has advanced past the main function. The check for `MainFuncDef` will likely fail or be skipped because the tokens have already been consumed by `FuncDef`.
   
   More importantly, `FuncDef` does not distinguish between a regular function and `main`. It just parses a function. The grammar requires `MainFuncDef` to be parsed specifically at the end. The current code tries to parse `main` as a `FuncDef`. While structurally similar, the output requirements differ (`<MainFuncDef>` vs `<FuncDef>`). If `FuncDef` parses `main`, it outputs `<FuncDef>`, which is incorrect according to the expected output `<MainFuncDef>`.

   Additionally, the loop condition `while((isConst() || isFuncType()) && !end())` might terminate prematurely or incorrectly handle the transition to `MainFuncDef` if `MainFuncDef` isn't handled as a distinct case before being consumed as a `FuncDef`.

   The most direct error is that `main` is being parsed as a `FuncDef` instead of `MainFuncDef`. The code attempts to check for `MainFuncDef` but only *after* potentially consuming it as a `FuncDef` or in a way that doesn't prevent `FuncDef` from consuming it first.
