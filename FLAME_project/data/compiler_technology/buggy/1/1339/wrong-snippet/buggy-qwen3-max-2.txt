Test Case Input:

Expected Output: `1 j`, `3 j`, `5 j`
Actual Output: `1 j`, `5 j`

The issue is that the parser is not detecting the missing closing parenthesis on line 3 in the function call `f(a + 10;`.

Looking at the code, the problem appears to be in the `parseStmt()` method in `Parser.java`. When parsing a function call like `f(a + 10;`, the parser should detect that there's no closing parenthesis before the semicolon.

In the `parseStmt()` method, there's a section that handles assignment statements with function calls:

```java
if (tokens.get().isOf(TokenType.GETINTTK, TokenType.GETCHARTK)) {
    Token get = tokens.consume();
    tokens.consumeExpect(TokenType.LPARENT);
    Logger.logger.check(tokens.consumeExpect(TokenType.RPARENT), new ErrorItem(tokens.getBefore().getLineNum(), "j", "Stmt中缺少)"));
    Logger.logger.check(tokens.consumeExpect(TokenType.SEMICN), new ErrorItem(tokens.getBefore().getLineNum(), "i", "Stmt中缺少;"));
    return new GetStmt(lVal, get);
}
```

However, this only handles `getint()` and `getchar()` calls. For general function calls like `f(a + 10)`, the parser treats it as an `AssignStmt` where the right-hand side is an expression. The function call `f(a + 10` would be parsed as part of the expression, but the missing closing parenthesis isn't being detected properly.

Looking more closely at the `parseUnaryExp()` method, which handles function calls:


The issue is in the logic that checks for `TokenType.ASSIGN`. In the test case `f(a + 10;`, after parsing the expression `a + 10`, the next token is `SEMICN` (semicolon), not `ASSIGN`. So the condition `if (tokens.get().isOf(TokenType.ASSIGN))` is false, and it goes to the else branch where it tries to parse `FuncRParams` and then expects a closing parenthesis.

However, the problem is that when `parseFuncRParams()` is called, it will consume tokens until it can't form a valid parameter list, but it doesn't handle the case where the closing parenthesis is missing and instead encounters a semicolon.

Looking at the `parseFuncRParams()` method:

```java
// FuncRParams → Exp { ',' Exp }
private FuncRParams parseFuncRParams() {
    ArrayList<Exp> exps = new ArrayList<>();
    do {
        exps.add(parseExp());
    } while (tokens.consumeExpect(TokenType.COMMA) != null);
    return new FuncRParams(exps);
}
```

This method doesn't check for the closing parenthesis - it just keeps parsing expressions separated by commas until it can't. Then back in `parseUnaryExp()`, it tries to consume the closing parenthesis with `Logger.logger.check(tokens.consumeExpect(TokenType.RPARENT), ...)`, which should detect the missing parenthesis.

1. `parseStmt()` sees `f` (IDENFR) followed by `(`, so it parses `LVal` as just `f`
2. Then it sees `=`? No, there's no `=` in `f(a + 10;` - this is actually a function call statement, not an assignment!

Ah, that's the real issue! The input `f(a + 10;` is a function call statement, not an assignment. But the grammar shows that function calls can only appear as part of expressions, and standalone function calls should be parsed as `ExpStmt` (expression statements).

So `f(a + 10;` should be parsed as an `ExpStmt` containing an expression that is a function call.

In `parseStmt()`, the default case handles this:

So `tryParser.tryParseExp()` should successfully parse `f(a + 10` as an expression (function call), then it calls `parseExp()` which calls `parseAddExp()`, etc., eventually calling `parseUnaryExp()` for the function call.

In `parseUnaryExp()`, when it tries to parse `f(a + 10`, it will:
1. Consume `f` and `(`
2. Parse `FuncRParams` which parses `a + 10` as an expression
3. Then try to consume `)` but finds `;` instead
4. This should trigger the error check: `Logger.logger.check(tokens.consumeExpect(TokenType.RPARENT), new ErrorItem(tokens.getBefore().getLineNum(), "j", "UnaryExp中缺少)"));`

The `tokens.getBefore().getLineNum()` should give the line number of the last consumed token, which would be the `10` token on line 3.
