**Analysis of Error Locations:**

1.  **Error Type 'a' (Illegal `&` or `|`):**
    The `Lexer.java` handles `&` and `|`. In `tokenizeSeparator`, if it encounters `&` not followed by `&`, it creates an `ErrorToken` and sets `this.error`. However, the `Compiler.java` checks `lexer.getError()`. If this is set, it writes to `error.txt`. But looking at `Lexer.tokenizeSeparator`:
    ```java
    if (c == '&') {
        if (next == '&') {
            tokens.add(new Token(TokenType.AND, "&&", curLine));
        } else {
            error = new ErrorToken(TokenType.AND, "&&", curLine); // Sets error
            processor.backward();
        }
    }
    ```
    The issue here is that `tokenizeSeparator` continues to execute subsequent `if` blocks for `<`, `>`, `=` etc. because they are not `else if`. More critically, after detecting an error like `&`, the lexer might still try to process the next character or get into an inconsistent state if `processor.backward()` isn't handled perfectly in conjunction with the main loop. However, the bigger issue is likely in how the parser interacts with these tokens or if the lexer crashes.

    The `while` condition checks `Character.isDigit(c)`. `c` is the character *before* the loop starts. Inside the loop, `c` is updated. If the next char is not a digit, the loop terminates. Then `processor.backward()` is called. This seems okay.
