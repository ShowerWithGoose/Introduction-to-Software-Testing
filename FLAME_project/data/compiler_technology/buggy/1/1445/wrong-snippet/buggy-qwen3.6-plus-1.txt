Based on the comparison between the buggy output and the expected output, the error lies in the `parseUnaryOp` method.

**Analysis:**
1.  **Expected Output:** When parsing `-5`, the expected output includes `<UnaryOp>` after printing the minus sign (`MINU -`).
    ```text
    MINU -
    <UnaryOp>
    INTCON 5
    ...
    ```
2.  **Buggy Output:** The buggy output misses the `<UnaryOp>` tag.
    ```text
    MINU -
    <Number>   <-- This is also suspicious, see below
    INTCON 5
    ...
    ```
3.  **Code Inspection:** In `frontend/Parser/Parser.java`, the `parseUnaryOp` method is defined as:
    ```java
    public Node parseUnaryOp() {
        // UnaryOp → '+' | '−' | '!'
        ArrayList<Node> children = new ArrayList<>();
        children.add(NodeCreator.createNode(curToken)); read();
        return NodeCreator.createNode(SyntaxVarType.NUMBER, children); // ERROR HERE
    }
    ```
    It incorrectly returns a node with type `SyntaxVarType.NUMBER` instead of `SyntaxVarType.UNARY_OP`. Although `NodeCreator.createNode` prints the syntax component name *before* creating the node, the `parseUnaryExp` method calls `parseUnaryOp()` but does not explicitly print `<UnaryOp>` itself (unlike how `AddExp` or `MulExp` might handle intermediate nodes if they were structured differently, but here the responsibility lies with the specific parser method to trigger the print via `NodeCreator` or explicit print).
    
    Wait, looking closer at `NodeCreator.createNode`:
    ```java
    public static Node createNode(SyntaxVarType type, ArrayList<Node> children){
        Printer.printSynVarType(type); // This prints the tag
        switch (type) { ... }
    }
    ```
    So, if `parseUnaryOp` calls `NodeCreator.createNode(SyntaxVarType.NUMBER, ...)`, it will print `<Number>`.

    ```text
    MINU -
    <Number>      <-- Printed by parseUnaryOp calling createNode with NUMBER
    INTCON 5
    <Number>      <-- Printed by parseNumber calling createNode with NUMBER
    <PrimaryExp>
    <UnaryExp>

    ```
    
    The Expected Output for `-5`:
    ```text
    MINU -
    <UnaryOp>     <-- Should be printed here
    INTCON 5
    <Number>
    <PrimaryExp>
    <UnaryExp>
    <UnaryExp>
    ```

    The bug is that `parseUnaryOp` creates a node with type `SyntaxVarType.NUMBER` instead of `SyntaxVarType.UNARY_OP`. Consequently, it prints `<Number>` instead of `<UnaryOp>`. Additionally, logically, a UnaryOp node should be of type `UNARY_OP`.
