    However, the most obvious bug that causes "Wrong Answer" in these parsers is usually related to **missing outputs** or **incorrect error detection**.
    
    Let's look at `parserBlockItem` again.
    ```java
    public void parserBlockItem() {
      if (returnContent().equals("const")||returnContent().equals("int")
      ||returnContent().equals("char")) {
          parserDecl();
      } else if(returnContent().equals("}")) {
          return;
      }
      else {
          parserStmt();
      }
    }
    ```
    
    If `parserDecl` is called for `int main...` inside a block (which is invalid C, but `main` is only global), it might behave weirdly. But `main` is not inside a block.
    
    **Actually, look at `parserDecl` in `Grammar.java`**:
    
    ```java
    public void parserDecl(){
      if (this.now().getContent().equals("const")) {
          // ...
      } 
      else {
          this.move();
          if(returnContent().equals("main")) {
              this.up();
              return;
          }
          this.up();
          match_BType();
          parserVarDef();
          // ...
      }
    }
    ```
    
    If `parserDecl` is called when the current token is `int` and the next is `main`, it moves to `main`, sees `main`, moves back, and **returns without consuming anything**.
    
    In `parserCompUnit`, if the lookahead logic fails to identify `main` as a FuncDef/MainFuncDef start and calls `parserDecl`, `parserDecl` will return without advancing. This causes an **infinite loop** in `parserCompUnit`'s `while` loop because `returnContent()` remains `int`.
    
    Let's verify if `parserCompUnit` can call `parserDecl` for `int main`.
    
    Loop 1:
    `int`. `move` -> `main`. `move` -> `(`.
    `returnContent` is `(`.
    Condition: `[` `=` `;` `,` or `line > line`.
    `(` is none of these. `line` is likely same.
    So `parserDecl` is NOT called. `sign_1` remains 1.
    Loop continues? `returnContent` is `(`. Loop condition `int/char/const/,` is false. Loop ends.
    `if (sign_1 == 1) { up(); up(); }`. Cursor back to `int`.
    
    Loop 2:
    `int`. `move` -> `main`.
    `if(main)` -> true. `up`. `break`.
    `parserMainFuncDef`.
    
    So `parserDecl` is NOT called for `int main`.
    
    **However**, what if there is a `void` function?
    `void func() {}`.
    Loop 1: `void` is not `int/char/const/,`. Loop 1 skipped.
    Loop 2: `void`.
    `if(void)` -> `parserFuncDef`.
    
    Inside `parserFuncDef`:
    `move` -> `func`.
    `if(main)` -> false.
    `up` -> `void`.
    `match_funcType` -> consumes `void`.
    `match_Ident` -> consumes `func`.
    `match_lparent` -> consumes `(`.
    `parserFuncParams`.
    `match_rparent` -> consumes `)`.
    `parserBlock`.
    
    This seems correct.
    
    **Conclusion:**
    
    The code is complex and fragile. The most likely error causing a "Wrong Answer" (especially if the output is completely empty or missing parts) is in the **`output.java`** file where it checks `sign == 0`.
    
    In `Compiler.java`:
    ```java
    Grammar grammar  = new Grammar(token,list,sign,errors);
    output out1 = new output(list,error_flag,errors);
    ```
    
    `sign` is a local variable initialized to 0. It is NEVER updated by `Grammar`.
    `Grammar` has its own `sign` member which is updated to 1 on error.
    But `Compiler` passes the local `sign` (0) to `output`.
    
    So `output` always receives `sign = 0`.
    
    If `errors` is empty, it outputs `parser.txt`.
    If `errors` is NOT empty, it skips `parser.txt` and outputs `error.txt`.
    
    If the parser correctly parses the input, `errors` is empty. Output is generated.
    
    If the parser INCORRECTLY detects an error (adds to `errors`), then `parser.txt` is suppressed.
    
    Given the test case is valid, if the parser adds any spurious error, the output will be empty (or just error.txt).
