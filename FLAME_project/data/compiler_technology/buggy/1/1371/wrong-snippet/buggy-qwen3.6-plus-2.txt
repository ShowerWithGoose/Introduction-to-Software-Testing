        Let's look at `lexer.java` and comment handling.
        The input has multi-line comments `/* ... */`.
        The lexer handles `sign2` (multi-line comment state) passed from `Compiler`.
        
        In `Compiler.java`:
        ```java
        int sign_1 = 0;
        // ...
        while ((nextline = reader.readLine())!=null) {
            lexer analyze = new lexer (nextline,line,token,sign_1,errors);

            } else if (analyze.getSign1() ==  -1) {

        ```
        
        In `lexer.java`:
        ```java

            // ...

                 pos++;
                 // ...
            }
            // ...
        ```

        The `lexer` constructor modifies `sign1` (local to lexer instance) but returns it via `getSign1()`.
        The `Compiler` updates its `sign_1` based on `getSign1()`.
        
        If a line starts with `/*`, `sign1` becomes 1. `Compiler` sets `sign_1 = 1`.
        Next line, `sign2` is 1. Lexer skips until `*`.
        If it finds `*/`, it sets `sign1 = -1`?
        
        Look at `lexer` logic for `*/`:
        ```java
        if(now=='*') {
            pos++;
            if (input.charAt(pos)=='/') {
                sign1 = 0; // Wait, sets to 0?
                pos++;
                break;
            }
        }
        ```
        And later:
        ```java
        if(sign1==1&&input.charAt(pos-1)=='*'&&input.charAt(pos)=='/'){
             sign1 = 0;
             pos++;
        }
        ```
        
        The `Compiler` expects `getSign1()` to return `1` if still in comment, `-1` if comment ended?
        ```java
        if(analyze.getSign1() == 1) {
            sign_1 = 1;
        } else if (analyze.getSign1() ==  -1) {
            sign_1 = 0;
        }
        ```
        
        But `lexer` sets `sign1 = 0` when comment ends. It never sets `sign1 = -1`.
        So `Compiler`'s `sign_1` will remain `1` forever once a multi-line comment starts!
        
        Consequently, for all subsequent lines, `sign2` passed to `lexer` is `1`.
        The lexer will skip characters looking for `*`.
        If the rest of the file doesn't have `*` at the start of the remaining content in a way that resets it properly, or if the logic is flawed, it might skip valid code tokens.
        
        Specifically, if `sign1` is never `-1`, `sign_1` in Compiler stays `1`.
        Next line: `lexer` called with `sign2=1`.
        It skips until `*`.
        If the line is `int main() {`, there is no `*`.
        `while(sign2 == 1 && input.charAt(pos)!='*')` will increment `pos` until end of line.
        `if(pos>=input.length()) break;`
        No tokens added for this line!
        
        This explains why the output is empty or missing most tokens. The lexer swallows all code after the first multi-line comment because `sign1` is not set to `-1` to signal the compiler to reset `sign_1` to `0`.

        The fix is in `lexer.java`. When a multi-line comment ends (`*/` is found), `sign1` should be set to `-1` so that `Compiler` can reset `sign_1` to `0`.
