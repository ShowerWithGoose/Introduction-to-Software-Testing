Based on the problem description and the test case analysis:

1.  **Test Case Input**:
    ```text
    void f(int a {} // error 1 (Missing ')')
    
    int main( { // error 2 (Missing ')')
        int a = 10;
        f(a + 10; // error 3 (Missing ')')
        return 0;
    }
    ```

2.  **Expected Output**:
    ```text
    1 j
    3 j
    5 j
    ```
    Note: Line numbers in the input file correspond to:
    - Line 1: `void f(int a {}` -> Missing `)` after `a`. The last non-terminal before the expected `)` is likely part of the parameter list or the parameter itself.
    - Line 3: `int main( {` -> This is actually Line 3 in the provided input block if we count empty lines? Let's re-read the input carefully.
      Input:
      Line 1: `void f(int a {} // error 1`
      Line 2: (Empty)
      Line 3: `int main( { // error 2`
      Line 4: `    int a = 10;`
      Line 5: `    f(a + 10; // error 3`
      Line 6: `    return 0;`
      Line 7: `}`
      
      Wait, the expected output says `1 j`, `3 j`, `5 j`.
      - Error at Line 1: `void f(int a {}`. The parser expects `)` after `int a`. It encounters `{`. So it reports error `j` at line 1.
      - Error at Line 3: `int main( {`. The parser expects `)` after `(`. It encounters `{`. So it reports error `j` at line 3.
      - Error at Line 5: `f(a + 10;`. The parser expects `)` after `10`. It encounters `;`. So it reports error `j` at line 5.

3.  **Buggy Output**:
    ```text
    1 NONE
    1 j
    3 j
    5 j
    ```
    The buggy output contains an extra line `1 NONE`. This indicates that an error of type `NONE` is being logged and printed for line 1.

4.  **Code Analysis**:
    Looking at `FuncDef.java`:
    ```java
    public static Node parse() {
        FuncDef funcDef = new FuncDef();
        funcDef.parseAndAdd(NodeRegistry.FUNC_TYPE);
        funcDef.terminalCheck(LexType.IDENFR);
        funcDef.terminalCheck(LexType.LPARENT);
        switch (Lexer.INSTANCE.getLexType()) {
            case INTTK, CHARTK -> {
                funcDef.parseAndAdd(NodeRegistry.TERMINAL);
                funcDef.parseAndAdd(NodeRegistry.FUNC_F_PARAMS);
                funcDef.terminalCheck(LexType.RPARENT, ErrorType.j);
            }
            case RPARENT -> funcDef.parseAndAdd(NodeRegistry.TERMINAL);
            default -> processError(ErrorType.j); // <--- This handles missing params AND missing ')' if next token is not RPARENT
        }
        funcDef.parseAndAdd(NodeRegistry.BLOCK);
        return funcDef;
    }
    ```
    
    And `BType.java`:
    ```java
    public static Node parse() {
        BType bType = new BType();
        if (Lexer.INSTANCE.getLexType() == LexType.INTTK ||
            Lexer.INSTANCE.getLexType() == LexType.CHARTK) {
            bType.parseAndAdd(NodeRegistry.TERMINAL);
        } else {
            processError(ErrorType.NONE); // <--- This looks suspicious
        }
        return bType;
    }
    ```

    Let's trace `void f(int a {}`:
    1. `FuncDef.parse()` is called.
    2. `FUNC_TYPE` parses `void`.
    3. `terminalCheck(IDENFR)` parses `f`.
    4. `terminalCheck(LPARENT)` parses `(`.
    5. Current token is `int` (INTTK).
    6. Switch case `INTTK, CHARTK` matches.
    7. `parseAndAdd(TERMINAL)` consumes `int`.
    8. `parseAndAdd(FUNC_F_PARAMS)` is called.
       - `FuncFParams.parse()` calls `FuncFParam.parse()`.
         - `FuncFParam.parse()` calls `BType.parse()`.
           - `BType.parse()` sees `INTTK`. Matches. Consumes `int`? No, `TERMINAL` was already consumed in step 7?
           
           Wait, let's look at `FuncDef` again.
           `funcDef.parseAndAdd(NodeRegistry.TERMINAL);` consumes `int`.
           Then `funcDef.parseAndAdd(NodeRegistry.FUNC_F_PARAMS);`.
           
           Inside `FuncFParams.parse()`:
           `funcFParams.parseAndAdd(NodeRegistry.FUNC_F_PARAM);`
           
           Inside `FuncFParam.parse()`:
           `funcFParam.parseAndAdd(NodeRegistry.B_TYPE);`
           
           Inside `BType.parse()`:
           Checks if current lexType is INTTK or CHARTK.
           The current token is now `a` (IDENFR), because `int` was consumed by `FuncDef`.
           So `BType.parse()` goes to `else` branch: `processError(ErrorType.NONE);`.
           This generates the `1 NONE` error.
           
           Then `FuncFParam` continues: `terminalCheck(IDENFR)` consumes `a`.
           Then checks for `[`. Not present.
           Returns.
           
           Back in `FuncFParams`: Loop checks for `,`. Next token is `{`. Loop ends.
           Returns.
           
           Back in `FuncDef`:
           `terminalCheck(RPARENT, ErrorType.j)` is called.
           Current token is `{`. Expected `)`.
           Reports error `j` at line 1.
           
           So we get `1 NONE` and `1 j`.

    The issue is in `FuncDef.java`. It manually consumes the `BType` token (`int`/`char`) using `TERMINAL` before calling `FUNC_F_PARAMS`. However, `FUNC_F_PARAMS` -> `FUNC_F_PARAM` -> `B_TYPE` expects to parse the `BType` itself. By consuming it prematurely, `B_TYPE` fails to find the expected token and logs a `NONE` error.

    The correct approach in `FuncDef` should be to let `FUNC_F_PARAMS` handle the parsing of parameters completely, including their types. Or, if `FuncDef` wants to peek, it shouldn't consume. But here it consumes.
    
    Actually, looking at the grammar:
    `FuncFParam → BType Ident ['[' ']']`
    
    In `FuncDef`:
    ```java
    case INTTK, CHARTK -> {
        funcDef.parseAndAdd(NodeRegistry.TERMINAL); // Consumes 'int' or 'char'
        funcDef.parseAndAdd(NodeRegistry.FUNC_F_PARAMS); // Expects to parse BType again?
    ...
    ```
    
    If `FuncFParams` expects to start with a `BType`, then `FuncDef` should NOT consume the `BType` token. The check `case INTTK, CHARTK` is just to decide whether to enter the parameter parsing block or go to `RPARENT`. It should not consume the token.

    Therefore, the error is in `FuncDef.java` where it incorrectly consumes the `BType` token.
