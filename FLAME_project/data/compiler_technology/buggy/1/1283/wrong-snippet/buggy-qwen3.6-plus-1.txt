Based on the comparison between the Buggy Output and the Expected Output, the discrepancy lies in the handling of the `printf` string constant.

**Buggy Output:**
```text
STRCON "%c%cn"
```

**Expected Output:**
```text
STRCON "%c%c\n"
```

The input source code contains: `printf("%c%c\n",b[i],str[i]);`.
The lexer is responsible for tokenizing this string. The expected output shows that the escape sequence `\n` should be preserved as part of the string content (`"%c%c\n"`). However, the buggy output shows `"%c%cn"`, indicating that the backslash `\` was consumed or ignored during lexical analysis, leaving only `n`.

Looking at `Lexer/Lexer.java`, in the section handling double-quoted strings (`StringConst`):

```java
        //双引号StringConst
        else if(ch=='\"'){
            token+="\"";
            if(!isEnd) ch=getChar();
            else return;
            while(!isEnd&&ch!='\"'){
                if(ch=='\\'){
                    if(!isEnd) ch=getChar();
                }
                token+=ch;
                ch=getChar();
            }
            token+="\"";
            tokens.add(new Token(Token.Type.STRCON,token,lineNum));
            next();
        }
```

The logic `if(ch=='\\'){ if(!isEnd) ch=getChar(); }` advances the character pointer `ch` to the next character (the escaped character, e.g., `n`) but **does not append the backslash `\` to the `token`**. It then proceeds to `token+=ch;`, which appends the `n`. This results in the loss of the backslash in the output token content.

To match the expected output where the raw string content including the backslash is preserved (or specifically that `\n` appears as `\n` in the output), the backslash itself needs to be added to the token before consuming the next character.
