The issue lies in how the parser handles errors. When a syntax error occurs (like a missing token), the parser currently just records the error but often fails to synchronize or stop parsing the current construct correctly, potentially leading to cascading errors or missed errors.

Specifically, look at `PrimaryExp`:
```java
    private static AstNode PrimaryExp() {
        AstNode node = new AstNode(AstType.PrimaryExp, now.line);
        if (now.type == singleType('(')) {
            pushToken(node, now); // (
            node.add(Exp());
            pushToken(node, now, singleType(')'), new ErrInfo(ErrType.j, node.lstLine));
        } ...
    }
```
When `pushToken` encounters a mismatch (expected `)` but found something else), it adds the error but **does not consume any token**. The `index` remains pointing to the incorrect token. In recursive descent parsers, if you don't consume or skip tokens after an error, the parser might get stuck or misinterpret subsequent tokens.

However, the more critical issue in "missing parenthesis" detection for complex expressions often relates to how the parser recovers. If `PrimaryExp` expects `)` and doesn't find it, it reports error `j`. But if the calling function (like `MulExp` or `AddExp`) continues to parse, it might consume tokens that were intended to be part of the outer structure, masking further errors or causing incorrect line numbers.
