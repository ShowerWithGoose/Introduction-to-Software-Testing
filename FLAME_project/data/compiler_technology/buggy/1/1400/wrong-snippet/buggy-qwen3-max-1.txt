Based on the test case and error output discrepancy, the issue lies in how the line numbers for missing semicolon errors (error type 'i') are being reported. The buggy output reports lines 4, 11, 13 while expected is 2, 9, 13.

Looking at the test input:
- Error 1: `printf("Hello SysY!\n")` missing semicolon on line 2
- Error 2: `int a` missing semicolon on line 9  
- Error 3: `return 0` missing semicolon on line 13

The problem is in the `parseExpStmt()` method in Parser.java. When a semicolon is missing, it uses `tokens.lastToken().getLine()` to determine the error line number. However, `lastToken()` returns the token before the current position, which may not be on the same line as where the semicolon should be.

Specifically, after parsing an expression that spans multiple tokens, the last token might be from an earlier line, causing incorrect line number reporting.

    public Node parseExpStmt() {
        ArrayList<Node> children = new ArrayList<>();
        if (curToken.getType() == TokenType.IDENFR || curToken.getType() == TokenType.PLUS
                || curToken.getType() == TokenType.MINU || curToken.getType() == TokenType.NOT
                || curToken.getType() == TokenType.LPARENT || curToken.getType() == TokenType.INTCON
                || curToken.getType() == TokenType.CHRCON) {
            children.add(parseExp());
        }
        if (curToken.getValue().equals(";")) {
            children.add(new Node(curToken));
            curToken = tokens.read();
        } else {
            Printer.addError("i", tokens.lastToken().getLine());
        }
        return new Node(children, SyntaxType.Stmt);
    }
