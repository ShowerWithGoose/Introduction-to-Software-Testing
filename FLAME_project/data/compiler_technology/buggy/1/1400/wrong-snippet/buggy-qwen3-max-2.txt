Based on the problem description and test case, the issue is with error line number reporting for missing semicolons (error type 'i'). The expected output shows errors on lines 2, 9, and 13, but the buggy output shows lines 4, 11, and 13.

Looking at the test input:
- Line 2: `printf("Hello SysY!\n") // error 1` - missing semicolon
- Line 9: `int a // error 2` - missing semicolon  
- Line 13: `return 0 // error 3` - missing semicolon

The problem appears to be in how the parser determines which line number to report when a semicolon is missing. The parser uses `tokens.lastToken().getLine()` to get the line number, but this may not correspond to the actual line where the statement began.

Examining the parser code, the issue is likely in the statement parsing methods where semicolon errors are reported. Specifically, in methods like `parseExpStmt()`, `parseAssignStmt()`, `parseReturnStmt()`, etc., the line number for the missing semicolon error is taken from `tokens.lastToken().getLine()`, which gives the line of the last successfully parsed token, not necessarily the line where the statement started.

            curToken = tokens.read();

        }
        return new Node(children, SyntaxType.Stmt);
    }

        //Exp的FIRST集：Ident,+-!,(,number,character

                || curToken.getType() == TokenType.MINU || curToken.getType() == TokenType.NOT
                || curToken.getType() == TokenType.LPARENT || curToken.getType() == TokenType.INTCON
                || curToken.getType() == TokenType.CHRCON) {
            children.add(parseExp());
        }
        if (curToken.getType() == TokenType.SEMICN) {
            children.add(new Node(curToken));
            curToken = tokens.read();
        } else {
            Printer.addError("i", tokens.lastToken().getLine());
        }
