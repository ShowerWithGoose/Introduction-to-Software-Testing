Based on the test case and error analysis:

1. The test input has missing semicolons at:
   - Line 2: `printf("Hello SysY!\n")` (missing semicolon)
   - Line 9: `int a` (missing semicolon)
   - Line 13: `return 0` (missing semicolon)

2. The buggy output shows errors at lines 2, 8, and 12, while expected output shows errors at lines 2, 9, and 13.

3. The issue appears to be in how the parser tracks line numbers when reporting missing semicolon errors (error code 'i'). The parser is using `tokenStream.get(p - 1).getLineNumber()` which may not correctly capture the actual line where the statement ends.

Looking at the parser code, the error reporting for missing semicolons consistently uses `tokenStream.get(p - 1).getLineNumber()`. However, when a semicolon is missing, the parser inserts a fake semicolon token with `tokenStream.get(p).getLineNumber()`, but still uses `p-1` for the error line.

The key issue is in the `parseExpStmt()` method and similar statement parsing methods where the error line number is determined.
