Based on the test case and output discrepancy, the issue is with error line number reporting for missing semicolons (error type 'i'). The expected output shows errors on lines 2, 9, and 13, but the buggy output shows lines 2, 8, and 12. This suggests the parser is using the wrong line number when reporting missing semicolon errors.

Looking at the parser code, in multiple statement parsing methods (like `parseAssignStmt`, `parseExpStmt`, etc.), when a semicolon is missing, the error is reported using `tokenStream.get(p - 1).getLineNumber()`. However, this gets the line number of the last token before the expected semicolon, which may not be the correct line where the statement ends.

In the test case:
1. `printf("Hello SysY!\n")` is on line 2, missing semicolon → should report line 2
2. `int a` is on line 9, missing semicolon → should report line 9  
3. `return 0` is on line 13, missing semicolon → should report line 13

The parser is likely using the line number of the last token in the expression (like the string constant, variable name, or number) rather than the actual line where the statement appears.
