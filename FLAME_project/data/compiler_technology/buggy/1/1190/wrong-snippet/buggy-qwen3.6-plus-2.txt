Based on the problem description, the test case input, and the expected output, here is the analysis:

**Summary:**
The program fails to detect "Missing semicolon" errors (error code `i`) in three specific locations within the `testfile.txt`:
1.  Line 2: `printf("Hello SysY!\n")` is missing a semicolon.
2.  Line 9: `int a` is missing a semicolon.
3.  Line 13: `return 0` is missing a semicolon.

The expected output is:
```
2 i
9 i
13 i
```
The buggy output is empty, meaning the parser is not generating these error reports.

**Error Location Analysis:**
The logic for detecting missing semicolons is implemented in `frontend/Parser.java`. Specifically, in methods like `parseConstDecl`, `parseVarDecl`, `parseStmt`, etc., there are checks like:
```java
if (match(Token.Type.SEMICN)) {
    // consume semicolon
} else {
    peekToken(-1).setErrorCode('i');
}
```
The method `peekToken(-1)` is intended to retrieve the last successfully parsed token to report the error line number. Let's look at the implementation of `peekToken` and `advance` in `frontend/Parser.java`.

In `advance()`:
```java
private void advance() {
    currentTokenIndex++;
    if (currentTokenIndex >= tokens.size()) {
        currentTokenIndex = tokens.size() - 1;
    }
    currentToken = tokens.get(currentTokenIndex);
}
```
When the end of the file is reached, `currentTokenIndex` stays at `tokens.size() - 1`.

In `peekToken(int number)`:
```java
private Token peekToken(int number) {
    if (currentTokenIndex + number >= tokens.size()) {
        return tokens.get(tokens.size() - 1);
    }
    if (currentTokenIndex + number < 0) {
        return tokens.get(0);
    }
    return tokens.get(currentTokenIndex + number);
}
```
When `peekToken(-1)` is called:
1.  If `currentTokenIndex` is 0, `currentTokenIndex + (-1)` is -1. The condition `currentTokenIndex + number < 0` is true, so it returns `tokens.get(0)`. This is incorrect if the error occurs later, but usually, we don't check for missing semicolons at the very first token.
2.  More importantly, consider the case where we have parsed some tokens and are at the end of the file or near it.
    *   Example: `return 0` (EOF).
    *   Tokens: `RETURNTK`, `INTCON`, `EOF` (or just end of list).
    *   Parser parses `return`, advances. Parses `0` (as Exp), advances.
    *   Now `currentToken` is likely the last token or an EOF marker if present. If the lexer doesn't produce an explicit EOF token, `currentToken` is the last valid token (`INTCON`).
    *   The parser checks `match(Token.Type.SEMICN)`. It fails.
    *   It calls `peekToken(-1)`.
    *   If `currentTokenIndex` points to the last token (e.g., index `N-1`), then `currentTokenIndex + (-1)` is `N-2`. This should return the token before the current one.
    *   However, let's look at how `parseExp` and underlying functions work. They consume tokens.
    *   In `parseStmt` for `return`:
        ```java
        case RETURNTK:
            stmt.setType(8);
            stmt.addToken(currentToken()); // 'return'
            advance();
            if (match(Token.Type.LPARENT) || ... ) {
                stmt.addExp(parseExp()); // Exp
            }
            if (match(Token.Type.SEMICN)) {
                stmt.addSemicn(currentToken()); // ';'
                advance();
            } else {
                peekToken(-1).setErrorCode('i');
            }
            break;
        ```
    *   After `parseExp()` finishes parsing `0`, `currentToken` will be the token *after* `0`. If there is no semicolon, it might be `}` or EOF.
    *   If it is `}`, `peekToken(-1)` should return `0` (the INTCON). The line number of `0` is 13. This seems correct.
    *   Why is the output empty?

I suspect the issue might be in **how the error code is set on the token**.
`peekToken(-1).setErrorCode('i')`.

If `peekToken(-1)` returns a token that is **not** the one stored in the AST node, the error will be lost.

In `printf` case:
Stored: `rParent` (which is `)`).
`peekToken(-1)` returns `)`.
Same object?
`tokens` list contains the token objects.

`peekToken` returns reference from `tokens`.
Yes, same object.

Stored: `exp` containing `Number` containing `intConst` (which is `0`).
`peekToken(-1)` returns `0`.
Same object?
Yes.

In `int a` case:
Stored: `decl` containing `varDecl` containing `varDef` containing `ident` (which is `a`).

Same object?
Yes.

There is one subtle case: **What if `peekToken(-1)` returns `null`?**
`tokens` list shouldn't contain nulls in the middle unless `handleSymbol` failed.
If `handleSymbol` returns null, `tokens` has null.
If `peekToken` accesses a null index, it returns null.
`null.setErrorCode('i')` -> NullPointerException.
Program would crash. It didn't.

I will bet on the `peekToken` implementation having an off-by-one error or boundary issue that causes it to return the wrong token or fail silently in a way I'm missing, OR the `error()` propagation is broken for specific nodes.
