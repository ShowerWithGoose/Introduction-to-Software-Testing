Based on the problem description and test case, the program should detect missing semicolons (error type 'i') on lines 2, 9, and 13 of the input file. The issue appears to be in the error reporting logic for missing semicolons.

Looking at the test input:
- Line 2: `printf("Hello SysY!\n")` is missing a semicolon
- Line 9: `int a` is missing a semicolon  
- Line 13: `return 0` is missing a semicolon

The parser correctly identifies these as missing semicolon errors but may be reporting the wrong line number. According to the specification, "The error line number corresponds to the line where the last non-terminal before the semicolon is located."

The issue is likely in how the parser determines which token's line number to use when reporting missing semicolon errors. In several places in the Parser.java file, when a semicolon is missing, it calls:

```java
recordError(getPreviousToken().getLineNumber(), "i");
```

However, `getPreviousToken()` returns the token immediately before the current position, which may not be the correct "last non-terminal before the semicolon" as required by the specification.

        SyntaxTreeNode constDeclNode = new SyntaxTreeNode("ConstDecl");

            advance(); // 跳过 'const'

            constDeclNode.addChild(new SyntaxTreeNode("COMMA", ","));
