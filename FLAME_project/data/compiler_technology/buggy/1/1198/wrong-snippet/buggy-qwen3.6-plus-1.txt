1.  **Problem Analysis**: The program needs to report syntax errors with their corresponding line numbers. The error type `i` indicates a missing semicolon. The line number reported should be the line where the last non-terminal symbol before the expected semicolon appears.
2.  **Test Case Analysis**:
    *   Input Line 2: `printf("Hello SysY!\n")` is missing a semicolon. The last token is `)` on line 2. Expected error: `2 i`.
    *   Input Line 9: `int a` is missing a semicolon. The last token is `a` (IDENFR) on line 9. Expected error: `9 i`.
    *   Input Line 13: `return 0` is missing a semicolon. The last token is `0` (INTCON) on line 13. Expected error: `13 i`.
    *   Buggy Output: `2 i`, `8 i`, `12 i`.
    *   Discrepancy: The errors for lines 9 and 13 are reported as 8 and 12 respectively. This suggests an off-by-one error in line number tracking, likely caused by the Lexer consuming the newline character `\n` immediately after the token and incrementing `line_num` before the Parser records the error. When the Parser detects the missing semicolon, it looks at `Lexer.outs.get(i - 1)`, but if the Lexer has already advanced past the newline associated with that token, the stored line number might be incorrect or the index `i-1` refers to a token whose line number was updated incorrectly.

            Wait, the whitespace logic is flawed.
            ```java

                ch = f.read();

                continue;

            ```
            If `ch` is `\n` (end of line 8):
            1. `Character.isWhitespace('\n')` is true.
            2. `ch = f.read()` reads the first character of line 9 (e.g., space or 'i').
            3. Checks if *that* character is `\n`. It is not.
            4. `line_num` is NOT incremented.
            5. Loop continues with `ch` being the first char of line 9.
            6. `line_num` is still 8.
            7. The token `int` on line 9 is processed with `line_num = 8`.

            Similarly, `return 0` on line 13 would be recorded with line number 12 if the previous line ended with a newline that was consumed by this flawed whitespace logic without incrementing `line_num`.

            The fix should be in `frontend/Lexer.java` in the whitespace handling block. It should increment `line_num` if the current character `ch` is `\n`, before reading the next character, or handle newlines specifically.
