Based on the problem description and the comparison between the buggy output (empty) and the expected output, the program fails to produce any output. This suggests that the parsing logic either crashes, enters an infinite loop, or fails to traverse the token list correctly.

Looking at the `Praser.java` file, specifically the `CompUnit` method:

```java
    private void CompUnit(TreeNode curNode){
        while(curIndex<tokenList.size()){
            if(match(curToken, "int") && match(tokenList.get(curIndex+1), "main")){
                curNode.addChild(new TreeNode(null , Component.MainFuncDef));
                MainFuncDef(curNode.getNextNode());
            }
            else if(match(tokenList.get(curIndex+2), "(")){
                curNode.addChild(new TreeNode(null , Component.FuncDef));
                FuncDef(curNode.getNextNode());
            }
            else if(match(curToken, "const") || match(curToken, "int") || match(curToken, "char")){
                curNode.addChild(new TreeNode(null , Component.Decl));
                Decl(curNode.getNextNode());
            }
        }
    }
```

The `while` loop condition is `curIndex < tokenList.size()`. However, inside the loop, if none of the `if/else if` conditions are met, or even if they are met but the called methods (like `MainFuncDef`) do not advance `curIndex` past the end of the construct in a way that eventually breaks the loop or matches the next component, the loop might become infinite or stuck.

More critically, let's look at `MainFuncDef`:

```java
    private void MainFuncDef(TreeNode curNode){
        if(match(curToken, "int")){
            curNode.addChild(new TreeNode(curToken, Component.Terminal));
            next();
        }
        if(match(curToken, "main")){
            curNode.addChild(new TreeNode(curToken, Component.Terminal));
            next();
        }
        if(match(curToken, "(")){
            curNode.addChild(new TreeNode(curToken, Component.Terminal));
            next();
            errorJ(curNode);
            // Block
            curNode.addChild(new TreeNode(null, Component.Block));
            Block(curNode.getNextNode());
        }
    }
```

In `CompUnit`, the check `match(tokenList.get(curIndex+1), "main")` accesses `curIndex+1`. If `curIndex` is at the last token, this will throw an `IndexOutOfBoundsException`. However, the input starts with `int main()`, so `curIndex` is 0, `curIndex+1` is 1 (`main`), which is valid.

The main issue lies in the `CompUnit` loop logic. It checks for `MainFuncDef` first. If it matches, it calls `MainFuncDef`. `MainFuncDef` parses `int`, `main`, `(`, `)`, and `Block`. After `MainFuncDef` returns, `curIndex` should be pointing to the token after the block (likely EOF or nothing). The `while` loop checks `curIndex < tokenList.size()`. If `MainFuncDef` consumed all tokens, `curIndex` equals `tokenList.size()`, and the loop terminates. This seems correct for a single `main` function.

However, let's look closer at `CompUnit`'s second condition: `else if(match(tokenList.get(curIndex+2), "("))`. This access `curIndex+2` is dangerous if there aren't enough tokens. But more importantly, if the first `if` condition `match(curToken, "int") && match(tokenList.get(curIndex+1), "main")` is true, it enters `MainFuncDef`.

Let's trace the input: `int main() { ... }`
1. `curIndex=0`, token=`int`. `match("int")` is true. `tokenList.get(1)` is `main`. `match("main")` is true.
2. Enters `MainFuncDef`.
3. Parses `int`, `main`, `(`, `)`, `Block`.
4. Inside `Block`, it parses `{`, then `BlockItem`s, then `}`.
5. After `Block` returns, `MainFuncDef` returns.
6. Back in `CompUnit`, `curIndex` is now at the end of the token list.
7. Loop condition `curIndex < tokenList.size()` becomes false. Loop ends.

This logic seems theoretically sound for a correct program ending with `MainFuncDef`. Why is the output empty?

    }

And `Token.print`:
```java
    public void print(FileWriter stdWriter) throws IOException {
        stdWriter.write(String.format("%s %s\n",type.toString(), word, lineNum));
    }
```

Wait, `String.format("%s %s\n", type.toString(), word, lineNum)` has 2 format specifiers `%s %s` but 3 arguments `type.toString()`, `word`, `lineNum`. This will throw a `MissingFormatArgumentException` or similar runtime exception inside the `try-catch` block in `TreeNode.print`. The exception is caught and ignored (`catch (Exception e) {}`), resulting in **no output** being written to the file.

The expected output format is `word category code word's character/string form`.
The `Token.print` method should likely be:
`stdWriter.write(String.format("%s %s\n", type.toString(), word));`
Or if line number is needed for debugging, it's not part of the specified output format for `parser.txt`. The specification says: `word category code word's character/string form (separated by a single space)`.

So the error is in `Token.java` in the `print` method.
