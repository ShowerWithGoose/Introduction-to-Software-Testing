2. Line 3: `int main( {` - Missing `)` after `main`. Expected error `3 j`.

   This explains the spurious `5 i` error. The parser detected the missing `)` in `parseUnaryExp` (outputting `5 j`), but did not advance past the error point sufficiently to skip the contents of the function call or align with the semicolon. Consequently, when control returned to `parseStmt`, it checked for a semicolon again, found `a` instead, and reported a missing semicolon `5 i`.

   The issue is that when a syntax error is detected and handled (like missing `)`), the parser should ideally synchronize or ensure that the subsequent check in the calling routine doesn't trigger another false positive error for the same location. Specifically, in `parseUnaryExp`, if `checkJ` fails, it means `)` is missing. The parser should probably attempt to synchronize or at least not leave the token stream in a state where the caller interprets the current token (which is part of the malformed argument list) as the start of a new statement component expecting a semicolon.

   However, a simpler fix often used in these recursive descent parsers with error recovery is that if an error is detected in a sub-component, we might need to consume tokens up to a synchronization point (like `)` or `;`) or simply accept that the error was reported and prevent double reporting.
   
   In this specific code structure, `parseUnaryExp` reports `j`. It does NOT consume the `)`. It leaves the pointer at `a`.

No, `j` is required.
