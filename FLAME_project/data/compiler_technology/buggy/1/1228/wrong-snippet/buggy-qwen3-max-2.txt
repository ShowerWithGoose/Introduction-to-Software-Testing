Based on the test case input and expected vs actual output, the issue is that an extra "5 i" (missing semicolon error) is being reported. Looking at the test input:

```
void f(int a {} // error 1
int main( { // error 2
    int a = 10;
    f(a + 10; // error 3
    return 0;
}
```

The errors should be:
- Line 1: missing ')' after parameter list → error j
- Line 3: missing ')' in function call → error j  
- Line 5: missing ')' in function call → error j

But the buggy output shows an additional "5 i" (missing semicolon on line 5).

Looking at the parser code, in the `parseStmt()` method, when handling function calls like `f(a + 10;`, the code processes it as an LVal assignment statement. After parsing the expression `a + 10`, it checks for a semicolon using `errorTool.checkI(token, line)`. However, at this point, the current token is likely `;` from the input `f(a + 10;`, but due to the missing closing parenthesis `)`, the parser's state is confused.

The real issue is in the `parseUnaryExp()` method when handling function calls. When parsing `f(a + 10;`, the parser expects a closing `)` but finds `;` instead. It reports error j correctly, but then continues parsing and later also reports error i for the same semicolon.

Looking more closely at the `parseStmt()` method in the LVal assignment branch, after handling getint/getchar/function calls, it always checks for a semicolon with `errorTool.checkI(token, line)`. However, in the case where there was already a parenthesis error, the semicolon check may be incorrectly applied.

The specific problem appears to be in the function call handling within `parseStmt()`. When processing `f(a + 10;`, after reporting the missing `)` error (j), the parser should not also report a missing semicolon (i) because the semicolon is actually present.
