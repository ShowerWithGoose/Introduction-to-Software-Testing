Based on the problem description and test case, the issue is that the error lines are being reported in the wrong order. The expected output shows errors on lines 2, 9, and 13 in that order, but the buggy output shows them as 13, 2, 9.

Looking at the test input:
- Line 2: `printf("Hello SysY!\n") // error 1` - missing semicolon
- Line 9: `int a // error 2` - missing semicolon  
- Line 13: `return 0 // error 3` - missing semicolon

The errors are being collected correctly but not sorted properly. In the `Compiler.java` file, the `dumpError()` method sorts the error lines using `Collections.sort(errorLines)`, but this sorts the string representations lexicographically rather than numerically by line number.

However, looking more closely at the `Error` class, it implements `Comparable<Error>` which should sort by line number first. The issue is in how the errors are being added to the list.

The real problem is in the parser's error reporting logic. When the parser encounters a missing semicolon error, it uses `getErrorLine()` which returns `tokens.get(cursor - 1).line`. However, the cursor position when detecting the missing semicolon may not correspond to the actual line where the statement began.

Looking at the `parseStmt()` method in `Parser.java`, specifically the part that handles the general `[Exp] ;` case (which covers the printf statement, variable declaration, and return statement), the error line is determined by `getErrorLine()` which gets the line of the previous token.

For the test case:
- The printf on line 2 ends without a semicolon, so when the parser expects SEMICN but finds something else, it reports the error line as the line of the last token in the printf call, which is line 2.
- The `int a` on line 9 is actually parsed as a `Decl` -> `VarDecl`, and in `parseVarDecl()`, when it doesn't find SEMICN, it calls `getErrorLine()` which should return line 9.
- The `return 0` on line 13 is handled in `parseStmt()`'s RETURNTK case, and when it doesn't find SEMICN, it reports the error line as 13.

The errors are being collected with the correct line numbers, but they're being added to the error list in the order they are encountered during parsing, which may not be the same as the source code line order.

However, the `Error` class does implement proper comparison by line number, and `dumpError()` calls `Collections.sort(errorLines)` on the string representations, not on the Error objects themselves.

Wait, looking at `dumpError()`:

```java
public static void dumpError() {
    String errFilePath = "error.txt";
    List<String> errorLines = new ArrayList<>();

    for (Error error: errors) {
        errorLines.add(error.toString());
    }
    Collections.sort(errorLines);
    // ...
}
```

The issue is that it's sorting the **string representations** (`"2 i"`, `"9 i"`, `"13 i"`) lexicographically, not the `Error` objects. Lexicographic sorting would put `"13 i"` before `"2 i"` because `'1'` < `'2'`.

The fix should be to sort the `Error` objects directly, not their string representations.
