Based on the problem description and test case, the issue is that error messages are not being output in ascending order by line number. The test case shows errors on lines 2, 9, and 13, but the buggy output shows them as "13 i", "2 i", "9 i" instead of the expected "2 i", "9 i", "13 i".

Looking at the code, the `dumpError()` method in `Compiler.java` sorts the error lines using `Collections.sort(errorLines)`. However, the `Error.toString()` method returns a string like "line type", and when these strings are sorted lexicographically, "13 i" comes before "2 i" because string comparison compares character by character ('1' < '2').

The issue is in the `Error` class's `compareTo` method. While it correctly compares by line number first, the `dumpError()` method converts errors to strings first and then sorts the strings, rather than sorting the `Error` objects directly.

However, looking more carefully at the `dumpError()` method:

The problem is that it's sorting the string representations instead of the `Error` objects themselves. It should sort the `errors` list first, then convert to strings.

The parser should be recording the correct line numbers (2, 9, 13) but it appears to be recording them incorrectly, or the sorting is happening on strings instead of numeric line numbers.
