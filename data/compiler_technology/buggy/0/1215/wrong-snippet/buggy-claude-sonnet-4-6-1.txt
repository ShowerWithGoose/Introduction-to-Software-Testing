## Summary

**Problem:** Lexical analyzer that tokenizes source code and detects error type `a` (single `&` or `|` appearing individually).

**Test failure:** The program fails to compile — `javac: not found` — meaning the Java source cannot be compiled in the test environment. This is a build/compilation error, not a runtime logic error.

**Root cause:** The project uses a package structure (`package frontEnd;`, `package frontEnd.lexer;`) but the compilation setup is broken. The most likely code-level issue causing compilation failure is in `Analzy.java`, which imports `import frontEnd.lexer.*;` from within the same package — a redundant self-import that isn't the core issue. More critically, `Symbol.java` defines `enum cCode` as package-private and `class Symbol` as public in the same file, but the file is named `Symbol.java`. This is valid Java. However, the real structural problem is that `Compiler.java` imports `import frontEnd.*;` but `CompileLexer` is in `frontEnd` package — that part is fine.

The actual compilation error is that `javac` is not found in the environment, but looking at the code itself for logical/semantic errors related to the error detection logic:

**The error detection logic is flawed** — when inside a block comment (`isComment==1`), `lastEnd` is never updated, causing the trailing-text check to always fire and write spurious errors.

The `lastEnd = matcher.end();` is inside the `if(isComment==0)` block. When a token is consumed while inside a comment (`isComment==1`), `lastEnd` is not advanced, so the trailing-text check at the end of the line incorrectly triggers error output.
