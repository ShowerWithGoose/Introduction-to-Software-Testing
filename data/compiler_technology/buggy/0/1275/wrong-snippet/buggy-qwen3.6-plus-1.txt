### Summary
The assignment requires a lexical analyzer that reads `testfile.txt`, identifies tokens according to a specified grammar, and outputs them to `lexer.txt`. If lexical errors occur, it should output the line number and error code to `error.txt`. The provided test case input (`/bin/sh: 1: javac: not found`) indicates a compilation environment issue rather than a program input, but analyzing the source code reveals clear logical and semantic flaws in `frontend/Lexer.java`:
1. **Missing `CharConst` Handling**: The token table explicitly defines `CharConst` (`CHRCON`), but the lexer completely lacks logic to parse single-quoted character constants (`'...'`).
2. **Incorrect Line Number for String Tokens**: When a string spans multiple lines, the code increments `lineNumber` inside the parsing loop but then uses the *updated* `lineNumber` (the line of the closing quote) for the token. Lexical analyzers should report the *starting* line number of the token.
3. **Static Map Initialization in Constructor**: `categoryCodes` is declared `static final` but populated inside the instance constructor. While it may run once, it violates Java best practices and can cause state leakage or initialization issues if multiple `Lexer` instances are created.
