## Summary

**Problem:** Lexical analyzer that tokenizes source code and detects errors (specifically single `&` or `|` characters).

**Test Case Failure:** The program fails to compile — `javac: not found` — meaning there's a build/compilation error in the Java source itself.

**Root Cause Area:** The compilation error is most likely caused by a missing `import` statement or a method/field reference that doesn't exist. Looking at the code, `TokenType` has no `isError` field initialized for non-error tokens in a way that would cause issues, but the more likely culprit is in `Lexer.java` where `token` could be `null` (if `nextToken()` returns `null`), causing a `NullPointerException` at runtime — but that's runtime, not compile-time.

The actual **compile-time** issue is in `Lexer.java`: the `TokenLize` method calls `token.getType().isError()` and `token.toString()` without null-checking, but more critically, `Token.toString()` overrides `Object.toString()` which is fine. However, looking more carefully — `TokenType` references `isError` as an instance field but it's never declared with a getter that would cause a compile error...

Actually, re-reading: the compile error is `javac: not found` — the JDK is not installed in the test environment. But assuming the grader expects compilable code, the logical/semantic errors are what matter.

The key bugs are:

1. **`Lexer.java` — `TokenLize`**: `nextToken()` can return `null` (no matching token found), but there's no null check before calling `token.getType()`.
2. **`SourceFile.java` — `readCurrentWord`**: The condition `!Character.isWhitespace(readCurrentChar()) && readCurrentChar() != '\n'` is redundant (whitespace includes `\n`), but more critically it reads the entire remaining word including operators, so multi-char tokens like `<=` get split incorrectly.
3. **`SourceFile.java` — `getLineNum`**: Returns 0-based line number, but error output expects 1-based line numbers.
