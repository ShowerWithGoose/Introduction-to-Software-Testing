**Summary**
- **Problem Description**: The lexical analyzer must output token information to `lexer.txt` for correct source programs, and error information (line number + error code) to `error.txt` for erroneous source programs.
- **Test Case Input**: `/bin/sh: 1: javac: not found` (This is an environment/compilation error from the grading system, but analyzing the source code reveals logical flaws that would cause test failures).
- **Buggy/Expected Output**: Not provided, but the specification explicitly requires conditional file routing based on whether lexical errors are detected.
- **Error Location**: 
  1. In `output.java`, the output file is hardcoded to `lexer.txt` regardless of the `sign` parameter. When `sign == 1` (indicating an erroneous program), it must write to `error.txt`.
  2. In `lexer.java`, multiple operator parsing blocks (`!`, `&`, `|`, `=`, `<`, `>`) incorrectly `break` out of the main loop when the operator appears at the end of a line. This causes the lexer to skip adding the corresponding single-character token or error token, leading to missing output or undetected errors.
