### Summary
- **Problem Description**: Implement a lexical analyzer that reads `testfile.txt`, identifies tokens (outputting type code and value to `lexer.txt`), or detects lexical errors (outputting line number and error code to `error.txt`). Only one error type `a` (individual `&` or `|`) is specified for erroneous inputs.
- **Test Case Input**: `/bin/sh: 1: javac: not found` (This indicates a compilation environment issue in the grading system, but we will analyze the source code for logical/semantic flaws).
- **Buggy/Expected Output**: Not provided, but static analysis reveals critical logical errors in the lexer's state management and comment stripping.
- **Identified Errors**:
  1. **Backtracking Logic Flaw (`pos = t`)**: In `Lexer.java`, when peeking ahead for two-character operators (`!=`, `&&`, `||`, `<=`, `>=`, `==`), the code uses `pos = t` to backtrack if the lookahead fails. If `getChar()` hits EOF, `pos` is incorrectly reset to the last character's index, causing `getNext()` to re-read it indefinitely (infinite loop) when the file ends with a single-character operator.
  2. **Comment Stripping Ignores String Literals**: `getCode()` strips `//` and `/* */` comments globally without checking if they appear inside string/char literals (e.g., `"a // b"` will be incorrectly truncated).
