**Summary of the Issue:**
The lexical analyzer contains a logical flaw in the handling of character constants. The main token regex `'(\\\\'|\\\\0|[^'\\\\])'` is too restrictive and does not account for common escape sequences like `\n`, `\t`, `\r`, etc. Consequently, tokens containing these escapes will either fail to match or be parsed incorrectly. Additionally, the `switch` statement checks for single characters (e.g., `case "n":`), but `characterValue` extracted via `substring` retains the leading backslash (e.g., `"\n"`), causing all escape sequence cases to fall through to `default`. The `case "'"` output format is also malformed. These issues will cause incorrect tokenization or output for character constants.
