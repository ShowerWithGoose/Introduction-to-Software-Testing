### Summary
- **Problem Description**: Implement a lexical analyzer that tokenizes source code into specified categories or detects lexical errors (specifically error `a` for standalone `&` or `|`). Outputs tokens to `lexer.txt` or errors to `error.txt`.
- **Test Case Input**: `/bin/sh: 1: javac: not found` (This indicates a compilation environment issue in the test runner, but static analysis of the provided source code reveals critical logical flaws that would cause runtime or incorrect output behavior).
- **Buggy/Expected Output**: Not provided, but code inspection shows clear deviations from lexical analysis rules.
- **Identified Errors**:
  1. **`frontend/tokenLexer/Lexer.java` -> `nextDigital()`**: The method incorrectly uses `patternIdentAll` (regex `[0-9a-zA-Z_]`) to consume subsequent characters. This causes letters and underscores following a digit to be incorrectly parsed as part of an `INTCON` token instead of stopping at the digit boundary. It should only match numeric digits.
  2. **`Compiler.java` -> Error Output Formatting**: The error output string lacks a trailing newline character (`\n`). This will cause multiple errors (if any) or subsequent outputs to be concatenated on the same line, violating standard line-based file formatting.
  3. **`frontend/tokenLexer/Lexer.java` -> `isSpecialStringNotIdent()`**: Missing boundary check before calling `codeFile.substring(pos + len, pos + len + 1)`. If a keyword appears at the very end of the file, this will throw an `IndexOutOfBoundsException`.
