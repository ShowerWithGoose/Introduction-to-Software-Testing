@@ [The loop iterates over TokenType.values() in an arbitrary order, which may cause shorter tokens (like "&" or "|") to be matched before longer valid tokens (like "&&" or "||"). However, the error detection for single "&" or "|" only triggers if such a single character is returned by matchRemainingLine. But because the enum order is not guaranteed to prioritize longer patterns first, a single "&" might be matched by the AND pattern's fallback in SrcFileLexer.matchRemainingLine (which returns "&" when pattern is "^&&" and input starts with "&"), and then incorrectly treated as an error even if it's part of a valid "&&". Conversely, if the single-character operators like PLUS or MINU come before multi-character ones, they may consume input prematurely. The core issue is that token matching must follow longest-match principle, but the current implementation relies on enum iteration order and ad-hoc fallbacks in matchRemainingLine, leading to incorrect tokenization or false error reporting.]
