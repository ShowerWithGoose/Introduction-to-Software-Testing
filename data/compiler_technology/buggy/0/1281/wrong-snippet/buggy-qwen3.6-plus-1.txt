### Summary of Information
- **Problem Description**: Implement a lexical analyzer that reads `testfile.txt`, tokenizes valid source code into `lexer.txt`, or detects lexical errors (specifically error type `a` for standalone `&` or `|`) and outputs them to `error.txt`.
- **Test Case Input**: `/bin/sh: 1: javac: not found` (This appears to be a system environment error from the grading platform rather than actual source code input).
- **Buggy Output / Expected Output**: Both are empty, likely due to the compilation/runtime failure caused by the environment error or logical bugs in the lexer.
- **Identified Logical/Semantic Errors**:
  1. **Missing Keywords**: The problem specification explicitly lists `char` → `CHARTK` and `getchar` → `GETCHARTK`. Both the `TokenType` enum and the `specificWords` map in `Lexer.java` are missing these entries. Consequently, `char` and `getchar` will be incorrectly tokenized as `IDENFR`.
  2. **Line Number Tracking Bug in Comments**: When processing single-line comments (`//`), the code consumes the newline character `\n` but fails to increment `lineNumber`. This causes all tokens following a `//` comment to report incorrect line numbers.
  3. **Error Recovery for `&` and `|`**: Using `backtrack()` after detecting a standalone `&` or `|` leaves the position pointing to the character *after* the operator, effectively skipping the erroneous character without properly consuming it. This can cause misalignment in subsequent tokenization.
