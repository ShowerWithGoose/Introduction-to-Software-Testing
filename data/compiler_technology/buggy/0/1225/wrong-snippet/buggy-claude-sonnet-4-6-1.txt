## Summary

**Problem:** The program fails to compile because `javac` is not found (`/bin/sh: 1: javac: not found`). This is a compilation error — the Java source files cannot be compiled.

**Root cause:** The `Token` class has a `getLineNo()` method that calls `Integer.parseInt(this.content)` unconditionally. However, `content` is not always a line number string — for normal tokens (e.g., `IDENFR`, `INTTK`, etc.), `content` holds the actual token text like `"main"` or `"123abc"`. This would cause a `NumberFormatException` at runtime, but more relevantly, the `Token` class is designed so that error tokens store the line number as a string in `content`, while normal tokens store the token text. The `getLineNo()` method is only safe to call on error tokens.

But the **actual compilation failure** (`javac: not found`) points to a build environment issue. Looking at the code itself for a **semantic/logical error** that would cause a compile-time error: the `TokenType` enum contains a lowercase member `a`, which is valid Java but unusual. The real issue visible in the code is in `Token.java`:

---

**Code snippet where the error is located:**

`frontend/lexer/Token.java`
```java
...
    public int getLineNo() {
        return Integer.parseInt(this.content);
    }
...
```

And in `frontend/lexer/Lexer.java`, the error token stores `lineNo` as a string but `Token` conflates content and line number in the same field:

`frontend/lexer/Lexer.java`

        Token token = new Token(type, content);
        tokenList.add(token);
        if (type == TokenType.a) {
            errTokenList.add(token);
        }

...
            // If we encounter an error
            if (isError) {
                addTokenToList(TokenType.a, String.valueOf(lineNo));

The error token is added to **both** `tokenList` and `errTokenList`. When the program is correct, `getTokenToPrint()` returns `tokenList`, which now contains the error token with `content = "1"` (the line number string). If `toString()` is called on it, it prints `a 1` — but more critically, the error token is mixed into the normal token list, corrupting the correct-program output. Additionally, `getLineNo()` will throw `NumberFormatException` for any non-error token if ever called on them.
