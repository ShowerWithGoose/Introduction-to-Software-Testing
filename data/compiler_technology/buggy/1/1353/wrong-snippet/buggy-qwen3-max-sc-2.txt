Based on the problem description and code review, the key issue appears to be in the error handling for missing semicolons (error type 'i'). The problem is that when a semicolon is missing, the error reporting mechanism uses `iterator.unRead(2)` to go back and determine the line number, but this may not correctly identify the line of the last non-terminal before the expected semicolon.

Looking at multiple parser files that handle semicolon errors (ConstDeclParser, VarDeclParser, StmtAssignParser, etc.), they all use the same pattern:

```java
private void handleIError() {
    iterator.unRead(2); // 后退两个以确定上一个非终结符的位置
    ErrorSet.getInstance().addError(new Error(ErrorType.MISSING_SEMICN, iterator.next().getLine()));
    this.semicn = null;
}
```

However, this approach is flawed because:
1. The number of tokens to unread may vary depending on the context
2. The last token before the missing semicolon might not be just 2 positions back
3. The line number should correspond to the line where the last non-terminal before the semicolon is located, but this method may not accurately capture that

The most representative location of this error is in the ConstDeclParser class:
