Based on the problem description and code review, the main issue appears to be in the error handling for missing semicolons (error type 'i'). The error reporting logic incorrectly determines the line number by moving back 2 tokens and then taking the next token's line number, which may not correspond to the actual location of the missing semicolon.

The most problematic implementation is in the `handleIError()` method in multiple parser classes. Let me show the key locations:

<frontend/parser/declaration/constant/ConstDeclParser.java>
...
    private void handleIError() {
        iterator.unRead(2); // 后退两个以确定上一个非终结符的位置
        ErrorSet.getInstance().addError(new Error(ErrorType.MISSING_SEMICN, iterator.next().getLine()));
        this.semicn = null;
    }
...

<frontend/parser/declaration/variable/VarDeclParser.java>
...
    private void handleIError() {
        iterator.unRead(2); // 后退两个以确定上一个非终结符的位置
        ErrorSet.getInstance().addError(new Error(ErrorType.MISSING_SEMICN, iterator.next().getLine()));
        this.semicn = null;
    }
...

<frontend/parser/statement/stmt/StmtAssignParser.java>
...
    private void handleIError() {
        iterator.unRead(2);
        ErrorSet.getInstance().addError(new Error(ErrorType.MISSING_SEMICN, iterator.next().getLine()));
        this.semicn = null;
    }
...

The error is that when a semicolon is missing, the parser should report the line number of the last token before where the semicolon should be, but the current implementation moves back 2 tokens and then reads forward 1 token, which may not give the correct line number for the error location.
