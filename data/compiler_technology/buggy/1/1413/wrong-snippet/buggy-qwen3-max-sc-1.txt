Based on the problem description and code review, the main issue appears to be in the error handling for missing semicolons (error type 'i'). The error line number should correspond to the line where the last non-terminal before the semicolon is located, but the current implementation often uses the wrong line number.

Looking at multiple parser classes, I can see a pattern where the errorLine is set incorrectly. For example, in StmtAssignParser, the errorLine is set to the line of the assignment operator (=) rather than the line of the last token before the expected semicolon.

The most problematic area is in the statement parsers where semicolon errors are handled. Let me show the key problematic snippets:
