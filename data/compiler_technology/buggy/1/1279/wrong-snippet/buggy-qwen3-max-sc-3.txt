Based on the problem description and code review, the main issue appears to be in the error handling for missing semicolons in the `parseConstDecl` and `parseVarDecl` methods. According to the problem specification, error type 'i' (missing semicolon) should report the line number of the last non-terminal before the semicolon, but the current implementation uses `getPreviousTokenLineNumber()` which may not correctly capture this.

Looking at the test case input showing a compilation error with "javac: not found", this suggests the actual test case isn't being run, but based on the code structure and common issues in recursive descent parsers, the most likely location of the error is in the semicolon error reporting logic.
