Based on the problem description and code review, the main issue appears to be in the error handling for missing semicolons (error type 'i'). The problem is that when a semicolon is missing, the code creates a fake SEMICN token but uses the wrong line number for the error report.

Looking at multiple parser files, I can see a consistent pattern where the error token is created with `first.my_line_num()` instead of the correct line number where the semicolon should appear. According to the problem description, "The error line number corresponds to the line where the last non-terminal before the semicolon is located."

The most clear example is in the VarDeclParser:

<frontend/parser/declaration/variable/VarDeclParser.java>
...
        if(!token.type.equals(TokenType.SEMICN))
        {
            this.iterator.unReadToken(1);
            this.semicn = new Token(TokenType.SEMICN,this.btype.token.linenum,";");
            this.error_tokens.add(new Token(TokenType.SEMICN, first.my_line_num(), "i"));
        } else{
            this.semicn = token;
        }
...

Similar issues exist in other parsers like ConstDeclParser, StmtAssignParser, etc., but the VarDeclParser shows the error most clearly where it uses `first.my_line_num()` for the error token instead of using the line number of the last parsed element before the expected semicolon.
