Based on the problem description and code review, the key issue appears to be in the error handling for missing semicolons (error type 'i'). The error line number should correspond to the line where the last non-terminal before the semicolon is located, but the current implementation uses the line number of the expression instead.

The most problematic code is in the statement parsers where semicolon errors are handled. Here's the relevant snippet:
