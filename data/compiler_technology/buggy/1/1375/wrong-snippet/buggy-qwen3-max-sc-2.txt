Based on the problem description and code review, the main issue appears to be in the error handling for missing semicolons (error type 'i'). The error line number should correspond to "the line where the last non-terminal before the semicolon is located", but the current implementation uses the line number of the expression instead.

Looking at the code, this pattern appears in multiple places where semicolon errors are handled. The most representative location is in the `StmtExpParser` class:

<frontend/parser/statement/stmt/StmtExpParser.java>
...
    public StmtExp parseStmtExp() {
        ExpParser expParser = new ExpParser(this.iterator,this.error_tokens);
        this.exp = expParser.parseExp();
        Token token = this.iterator.readNextToken();
        this.iterator.unReadToken(1);
        if(token.type.equals(TokenType.SEMICN)){
            this.semicn=this.iterator.readNextToken();
        }else{
            this.semicn=new Token(TokenType.SEMICN,exp.my_line_num(),";");
            this.error_tokens.add(new Token(TokenType.SEMICN,exp.my_line_num(),"i"));
        }
        StmtExp stmtExp = new StmtExp(this.exp, this.semicn);
        return stmtExp;
    }
...

Similar issues exist in other statement parsers like `StmtAssignParser`, `StmtBreakParser`, `StmtContinueParser`, etc., where the error line number is incorrectly set to the expression's line number rather than the line of the last non-terminal before the expected semicolon.
