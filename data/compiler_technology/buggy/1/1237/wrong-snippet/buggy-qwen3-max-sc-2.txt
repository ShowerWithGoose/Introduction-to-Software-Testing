Based on the problem description and code review, the main issue appears to be in the error handling logic for missing semicolons and brackets. The error reporting mechanism incorrectly determines the line number for errors by reading an extra token after unreading.

The most problematic code is in multiple parser files where error handling occurs. Here's the key problematic pattern:

<frontend/declaration/constant/ConstDeclParser.java>
...
        /* 处理i类错误：缺失; */
        this.semicn = token;
        if (!this.semicn.getType().equals(TokenType.SEMICN)) {
            // 回退两个 Token，获取分号前的非终结符行号
            this.iterator.unReadToken(2);
            Error error = new Error(this.iterator.readNextToken().getLineIndex(),
                    ErrorType.MISSING_SEMICN);
            ErrorTable.addError(error);
        }
...

Similar patterns exist in:
- VarDeclParser.java
- VarDefParser.java 
- LValParser.java
- PrimaryExpExpParser.java
- MainFuncDefParser.java
- StmtAssignParser.java
- StmtBreakParser.java
- StmtContinueParser.java
- StmtGetCharParser.java
- StmtGetIntParser.java
- StmtPrintParser.java
- StmtReturnParser.java

The issue is that after `unReadToken(2)`, calling `readNextToken()` gets the token before the expected semicolon/bracket, but this may not be the correct line number for the error according to the specification.
