Based on the problem description and code review, the main issue appears to be in the error handling for missing semicolons in statement parsing. According to the requirements, when a semicolon is missing, the error line number should correspond to "the line where the last non-terminal before the semicolon is located." However, in several places in the StmtParser.java file, the error line number is being set incorrectly.

The most problematic area is in the default case of the parseStmt() method where expression statements are handled. The code uses `semicolonExp.getLine() - 1` which may not correctly identify the line of the last non-terminal before the expected semicolon.
